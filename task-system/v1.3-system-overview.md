## 1. System Architecture

### The Hierarchy

```
STRATEGY LAYER (stable — changes weekly/monthly)
  ┌─────────────────────────────────────────────────────────────┐
  │  GOAL                                                        │
  │   └── Sub-Goal (own metric, own mechanism chain)             │
  │        └── Mechanism Chain (M1 → M2 → M3)                   │
  │             └── Mechanism Step (input → output @ ratio)      │
  │                  └── Path (interchangeable channel/tactic)   │
  ├─────────────────────────────────────────────────────────────┤
  │  BRIDGE: Path connects strategy to execution                 │
  ├─────────────────────────────────────────────────────────────┤
EXECUTION LAYER (volatile — changes daily)
  │                       └── Task (discrete work unit)          │
  │                            └── Subtask (atomic, one output)  │
  └─────────────────────────────────────────────────────────────┘
```

### How They Connect

| From | To | Relationship | Reference Field |
|------|----|-------------|-----------------|
| Goal | Sub-Goal | `HAS_SUBGOAL` | Goal file → Sub-Goals table |
| Sub-Goal | Mechanism Step | `HAS_MECHANISM` | Goal file → Mechanism Chain section |
| Mechanism Step | Mechanism Step | `CHAINS_TO` | Goal file → step ordering (M1→M2→M3) |
| Mechanism Step | Path | `FEEDS` (reverse) | Goal file → Paths table (`Feeds Step` column) |
| Path | Task | `CONTAINS_TASK` | Task file → `Goal:` field references PATH-ID |
| Task | Subtask | `HAS_SUBTASK` | Task file → Subtasks table |
| Goal | Foundation | `HAS_FOUNDATION` | Goal file → Foundations table |
| Foundation | Task | `IMPLEMENTED_BY` | Goal file → Foundations table (`Implementing Task(s)`) |
| Goal | Threshold Gate | `HAS_GATE` | Goal file → Threshold Gates table |
| Threshold Gate | Task/Path | `UNLOCKS` | Goal file → Threshold Gates table (`Unlocks`) |

### The 4-Hop Trace Rule

Every task must trace to a goal in ≤4 hops:

```
Task → Path → MechanismStep → SubGoal → Goal
```

If a task can't make this trace, it's orphaned — either link it or question its existence.

## 2. Foundations

The 8 irreducible infrastructure components every business needs. Derived from BMC/Lean Canvas synthesis.

| # | Foundation | What It Means | Example |
|---|-----------|--------------|---------|
| 1 | **Offer** | What you sell, to whom, packaged how | "12-week ADHD coaching program" |
| 2 | **Audience** | Who you serve + how to find them | "ADHD professionals, LinkedIn" |
| 3 | **Channel** | How you reach the audience | "LinkedIn outreach + content" |
| 4 | **Conversion Mechanism** | How interest becomes revenue | "Outreach → Qualify → Close" |
| 5 | **Delivery System** | How you deliver value after sale | "1:1 Zoom, structured program" |
| 6 | **Revenue Model** | How money flows in | "Monthly retainer via Stripe" |
| 7 | **Legal Framework** | Permission to operate | "Contract, T&Cs, GDPR" |
| 8 | **Cost Structure** | What it costs to operate | "Tools, time, subscriptions" |

### Where They Live
- **Defined in:** Goal file → `## Foundations` table
- **Each has:** Current State, Maturity (`raw → draft → tested → proven`), Sufficient? (✅/❌), Implementing Task(s)
- **Tracked by:** Weekly strategy review checks maturity changes
- **Relation to goals:** Every goal file assesses its own foundation state. Foundations are shared across goals (one "Offer" foundation node, referenced by multiple goals).

### Why They Matter
A missing foundation blocks the mechanism chain regardless of execution quality. If you have no offer, no amount of outreach converts. Foundations are infrastructure problems — they get priority unblock status.

## 3. Data Flow

### Layer-by-Layer Input/Output

| Layer | Input | Processing | Output |
|-------|-------|-----------|--------|
| **Goal** | Business objective, deadline | Decompose into sub-goals | Sub-goals with metrics, mechanism chains |
| **Sub-Goal** | Parent goal metric allocation | Define conversion funnel | Mechanism chain with steps and ratios |
| **Mechanism Step** | Upstream step output (or raw input for M1) | Convert at ratio | Downstream step input + throughput data |
| **Path** | Strategy decision ("use LinkedIn") | Execute via tasks | Throughput feeding mechanism step |
| **Task** | Path requirements, dependencies | Execute work | Verifiable outputs (artifacts, state changes) |
| **Subtask** | Parent task decomposition | Atomic work unit | Single verifiable output |

### Data Flow Between Layers

```
Goal metric (£5k/month)
  ↓ decompose
Sub-Goal metric (6 clients @ £500+)
  ↓ define funnel
Mechanism Chain ratios (5% CR accept, 20% qualify, 50% close)
  → implies required throughput at each step
  ↓ back-calculate
Required volume at M1 (need X contacts/week to hit goal)
  ↓ feeds
Path throughput targets (PATH-LI must deliver Y contacts/week)
  ↓ implemented by
Task outputs (LP-001 sources leads, LP-003 writes intros, etc.)
  ↓ verified by
Subtask outputs (each atomic deliverable)
```

### Feedback Flow (Bottom-Up)

```
Subtask output verified → Task completion gate passes
  ↓ aggregates
Path throughput measured → feeds mechanism step metrics
  ↓ aggregates
Mechanism step conversion ratios updated → bottleneck identified
  ↓ informs
Strategy review: is the bottleneck moving? Are paths working?
  ↓ informs
Goal review: is the goal achievable at current rates?
```

## 4. Strategy Layer Process

### 4.1 Create a Goal

**Input:** Business objective in Jamie's head
**Process:**
1. Copy `goal-template-v1.3.md` → `~/Master/goals/GOAL-XXX.md`
2. Fill: name, one-liner, metric (one number), measurement source, deadline
3. Set status: ACTIVE, mode: BUILDING
4. Link: `Strategy Change Protocol: strategy-change-protocol.md`

**Output:** Goal file with header fields populated.

### 4.2 Define Sub-Goals

**Input:** Goal metric
**Process:**
1. Decompose goal into independently trackable streams
2. Each sub-goal gets: ID, name, own metric, current value, target, status
3. Sub-goals partition the goal — their metrics should sum/combine to achieve the parent

**Output:** Sub-Goals table in goal file populated.

### 4.3 Set Up Mechanism Chain

**Input:** Sub-goal metric (e.g., "6 clients")
**Process:**
1. For each sub-goal, define the conversion funnel: what sequential steps turn raw input into the sub-goal metric?
2. Each step: Input label, Output label, Conversion Ratio, Current Throughput
3. Identify the **bottleneck** — step with worst ratio or lowest throughput
4. Back-calculate required M1 volume from goal target and chain ratios

**Output:** Mechanism Chain table per sub-goal. Bottleneck identified with action.

### 4.4 Define Paths

**Input:** Mechanism chain steps that need throughput
**Process:**
1. For each mechanism step, identify channels/tactics that can feed it
2. Each path: ID, name, which step it feeds, status, MVP flag
3. Designate ONE path as MVP — fewest requirements, fastest to revenue
4. Other paths: NOT STARTED or deferred behind threshold gates

**Output:** Paths table in goal file. MVP path identified.

### 4.5 Assess Foundations

**Input:** Goal requirements
**Process:**
1. For each of the 8 foundations, assess: what exists? what maturity? sufficient for this goal?
2. If insufficient: which task implements it? Create the task if it doesn't exist.
3. Insufficient foundations = infrastructure blockers → priority unblock

**Output:** Foundations table populated with current state, maturity, sufficiency, implementing tasks.

### 4.6 Set Threshold Gates

**Input:** Future work that shouldn't start yet
**Process:**
1. For each piece of deferred work: what quantitative condition must be true before starting?
2. Define: condition, metric, threshold value, current value, source (where to read it), what it unlocks
3. Gate status: LOCKED (below threshold) or UNLOCKED

**Output:** Threshold Gates table. Deferred Work table linked to gates.

### 4.7 Set System Stop Criteria

**Input:** Understanding of what "done building" means
**Process:**
1. Define checklist: what must all be true for the system to transition BUILDING → OPERATIONAL?
2. Typical criteria: all paths mapped, MVP path operational, foundations sufficient, bottleneck has active intervention
3. Mode transition rule: all criteria met + ≥2 weeks stable operation

**Output:** System Stop Criteria checklist in goal file.

## 5. Execution Layer Process

### 5.1 Create a Task

**Input:** Work needed by a path/foundation
**Process:**
1. Check `~/Master/TASK-REGISTRY.md` — does a profile for this task type exist?
   - YES → create instance (row in execution log)
   - NO → copy `template-v1.3.md` → `~/Master/tasks/[ID].md`
2. Determine task level: HEADLINES (minimal) or FULL SPEC (complex)
3. Fill required fields per level (see task-field-guide.md)
4. Classify owner: 🤖 Rich / 🔄 Hybrid / 🧑 Jamie / ❓ Undefined
5. Score Layer 0 (agent layer) → compute EBAM category
6. Set `Goal:` field to link: `GOAL-ID | PATH-ID | Mechanism Step`
7. Set `Location:` field: 🏠 Remote | 📍 On-site | 🔄 Either

**Output:** Task file created. Registry updated. Mirror pushed.

### 5.2 Link Task to Goal/Path

**Input:** Task file, goal file
**Process:**
1. Task `Goal:` field references: `GOAL-XXX | PATH-YY | MZ: StepName`
2. Goal file Paths table shows which tasks belong to which path
3. Verify 4-hop trace: Task → Path → MechanismStep → SubGoal → Goal

**Output:** Task traceable to goal. Bidirectional references.

### 5.3 Define Subtasks with Outputs

**Input:** Task work breakdown
**Process:**
1. Decompose task into atomic units — one output per subtask
2. Table format: `| # | Type | Task | Status | Owner | Output (proof of completion) |`
3. Each subtask gets its own owner (may differ from parent)
4. Note dependencies between subtasks ("needs #3")
5. Input validation: before starting subtask B, verify subtask A's output exists if B depends on A

**Output:** Subtasks table with every row having a defined, verifiable output.

### 5.4 Run the Completion Gate

**Input:** Task/subtask claiming to be done
**Process:**
1. **Check Win State Outputs:** For EACH output in the Outputs list:
   - Does it exist? (file created, column populated, script runs, state changed)
   - Can you verify it right now?
   - If ANY output doesn't exist → DO NOT mark done. Report what's missing.
2. **Check Subtask Outputs:** For each subtask:
   - Does the output exist?
   - If no output defined → subtask isn't properly specified. Flag it.
3. **Owner-specific rules:**
   - 🤖 Rich: Agent verifies all outputs. If all pass → mark done.
   - 🔄 Hybrid: Agent verifies Rich's outputs. Jamie's outputs need explicit "done"/"approved". Ask once, wait.
   - 🧑 Jamie: Only Jamie marks these done. Agent can ask but NEVER self-marks.
4. **If win state vague/missing:** Flag: "Win state has no verifiable outputs. Can't confirm completion."
5. **Post-completion:** Update task file status → COMPLETED. Update TASK-REGISTRY.md. Mirror to ~/clawd/. Push.

**Output:** Task verified complete with all outputs existing, or flagged with what's missing.

## 6. Tracing

### 6.1 Top-Down Trace (Strategy → Execution)

**Purpose:** "Given my goal, what should I work on?"

**Steps:**
1. Start at Goal → check status, mode
2. Read Sub-Goals → which is active? Which has worst metrics?
3. Read Mechanism Chain → where is the **bottleneck**? (worst ratio or lowest throughput)
4. Check Foundations → any insufficient? (infrastructure blocker = priority unblock)
5. From bottleneck step → which Paths feed it? Which is MVP?
6. From MVP Path → which Tasks are unblocked and highest priority?
7. From Task → which Subtasks are next?

**Dependency tracing during top-down:**
- Follow HARD dependencies (→) only for "what's physically blocking"
- Note SOFT dependencies (~>) separately
- Check THRESHOLD dependencies (⊕) against current values
- When you hit the root of a domain chain → check domain operational checklist
- If checklist item failing → THAT is the true root blocker

### 6.2 Bottom-Up Trace (Execution → Strategy)

**Purpose:** "Why am I doing this task? What goal does it serve?"

**Steps:**
1. Start at Task → read `Goal:` field
2. Identify Path → which mechanism step does it feed?
3. Identify Mechanism Step → which sub-goal owns it?
4. Identify Sub-Goal → which goal?
5. Verify: is this task serving the current bottleneck? If not, it may be misallocated effort.

### 6.3 The Strategy/Execution Boundary

| If the blocker is... | It's a... | Action | Who fixes |
|---------------------|-----------|--------|-----------|
| Task status (CAPTURED/BLOCKED/not started) | **Execution problem** | Do the task. Unblock it. | Daily execution |
| Conversion ratio (doing work but not converting) | **Strategy problem** | Wait for strategy review. Analyse mechanism step. | Weekly review |
| Missing foundation | **Infrastructure problem** | Priority unblock. | Immediate |
| Emotional resistance | **Behavioural problem** | Use intervention stack (EBAM). | Daily execution with interventions |

**Rule:** You cannot change strategy from execution mode. Capture insights as notes → act on them in the next strategy review.

## 7. Operational Processes

### 7.1 Morning Briefing (Daily — Execution Mode)

**What it reads:**
- Goal files (bottleneck, mode, foundations)
- Task files (status, priority, owner, dependencies)
- Task registry (what's active)
- H1/pipeline data (if applicable)

**What it outputs:**
```
## Context (auto-generated)
GOAL-001 [BUILDING]: Bottleneck at M2. 3/8 foundations sufficient.

## 🔴 Must Do
- [strategy emoji] [task] (🔓/⚙️/🚀 if hybrid) — [why]

## 🟡 Should Do
- [strategy emoji] [task] — [why]

## 🟢 Could Do
- [strategy emoji] [task] — [why]

## ⏸️ Explicitly NOT Doing
- [deferred items — active deferral, visible]
```

**The 3 Buckets (within each priority level):**

| Bucket | Icon | Meaning |
|--------|------|---------|
| Jamie Only | 🧑 | Physical, personal, undefined — Jamie must do |
| Rich E2E | 🤖 | Fully autonomous — Rich executes |
| Hybrid | 🔄 | Split work — with subtype: |
| | 🔓 | Unblock: Jamie does X so Rich can proceed |
| | ⚙️ | Systematise: Jamie creates spec so Rich automates forever |
| | 🚀 | Execute: Jamie directly produces output |

**Strategy Traffic Light (per task):**

| Emoji | Meaning |
|-------|---------|
| ✅ | **Aligned** — strategy sound, just execute |
| ⚠️ | **Suboptimal** — strategy has gaps but proceeding anyway |
| 🔴 | **Blocked** — something upstream broken, may be wasted effort |
| ❓ | **Undefined** — strategy not mapped for this task |

**Task Priority Traffic Light:**

| Colour | Criteria | Meaning |
|--------|----------|---------|
| 🔴 Red | max(criticality, strategic_leverage) = HIGH | Must do |
| 🟡 Amber | Moderate criticality or leverage | Should do |
| 🟢 Green | Low criticality + low leverage, or easy wins | Could do |

Within each colour, cognitive match (EBAM) determines order. Default: worst cognitive match first (hardest thing while energy is highest).

### 7.2 Weekly Review (Strategy Mode — OODA Observe + Orient + Decide)

**What it checks:**
1. Strategy sense-check: does the current strategy still make sense?
2. Data pull: update mechanism chain metrics (throughput, conversion ratios)
3. Volume check: enough data to judge? (minimum thresholds from strategy-change-protocol.md)
4. Mechanism chain review: has the bottleneck moved? Is current intervention working?
5. Signals table: run persevere vs change signals for each active path
6. Threshold gates: any approaching unlock? (≥80% = start preparing, 100% = unlock)
7. Deferred work: any unlocked items to activate?
8. Decision log: pending review dates this week?
9. Path archive: learnings from dropped paths?
10. Foundations: any maturity changes this week?

**Minimum Volume Thresholds (before judging):**

| Decision | Minimum Volume |
|----------|---------------|
| Is this outreach channel working? | ≥50 contacts made |
| Is our conversion rate real? | ≥20 conversations |
| Should we split a mechanism step? | ≥20 data points at that step |
| Is this path better than that path? | ≥30 contacts per path |
| Is the offer wrong? | ≥5 qualified prospects rejected |
| Should we add a new path? | MVP path operational + ≥4 weeks data |

**Stick-to-Plan Signals:**

| Persevere (execute harder) | Change the plan |
|---------------------------|-----------------|
| Leading indicators moving right direction | Leading indicators flat/declining after 2+ cycles |
| Haven't completed 2 full cycles | Surprising observations (expected → unexpected output) |
| Blocker is execution, not strategy | External environment changed |
| Doubts are emotional, not data-driven | Consistent feedback contradicts assumptions |
| Volume threshold not met yet | Bottleneck is structural, not effort-based |

**When to change strategy:** Only when signals-to-change outweigh signals-to-persevere AND volume thresholds are met. Follow the Change Protocol (strategy-change-protocol.md § The Change Protocol).

### 7.3 Monthly Goal Review

**What gets assessed:**
- [ ] Goal still valid? Has the objective, metric, or deadline changed meaning?
- [ ] Sub-goals balanced? All progressing, or one starving another?
- [ ] Mode transition? Should BUILDING → OPERATIONAL? (check System Stop Criteria)
- [ ] Environmental scan: world changed enough to invalidate strategy?
- [ ] Foundations reassessment: any "sufficient" now insufficient?
- [ ] Path portfolio: right number of paths? Too many (diluted) or too few (fragile)?
- [ ] Decision log review: were past decisions right in hindsight?

### 7.4 Quarterly Environmental Check

- [ ] Market conditions: new competitors, regulatory changes, economic shifts?
- [ ] Technology changes: new tools, platforms, AI capabilities?
- [ ] Personal circumstances: capacity, motivation, financial runway changed?
- [ ] Customer feedback patterns: still solving the right problems?

If any "yes" → schedule a Type 1 strategy review. Don't panic-pivot.

### 7.5 Threshold Gate Firing

**Who checks:** Scanner (automated via cron or manual during reviews)
**When:** Weekly during strategy review. Continuous via scanner if automated.
**What happens:**
1. Scanner reads gate's `source` field to get current metric value
2. Below threshold → report: "X at 3/10. 7 to go."
3. At ≥80% → alert: "X at 8/10 — start preparing."
4. At 100% → fire: change downstream status WAITING → CAPTURED. Alert Jamie. Log in Decision Log.

### 7.6 OODA Cadence Summary

| Layer | Cadence | Mode | OODA Phase |
|-------|---------|------|------------|
| Goal | Monthly | Strategy | Observe + Orient |
| Mechanism chain / bottleneck | Weekly | Strategy | Observe + Orient |
| Paths | Weekly | Strategy | Decide |
| Tasks | Daily | Execution | Act |
| Environment | Quarterly | Strategy | Orient |

## 8. Build → Run Transition

### Modes

| Mode | Meaning |
|------|---------|
| **BUILDING** | Active construction — creating infrastructure, defining processes, high cognitive load, frequent decisions |
| **OPERATIONAL** | Run mode — monitoring, exception handling, incremental optimisation. NOT structural changes |

### Transition Criteria (System Stop Criteria)

All of these must be true to transition BUILDING → OPERATIONAL:

- [ ] All paths mapped with tasks
- [ ] MVP path fully operational (all tasks IN PROGRESS or higher)
- [ ] Foundations sufficient (all 8 marked ✅)
- [ ] Bottleneck identified and has active intervention
- [ ] ≥2 weeks stable operation after above criteria met

### What Changes

| Aspect | BUILDING | OPERATIONAL |
|--------|----------|-------------|
| Time investment | High (active construction) | Low (<2hrs/day per goal) |
| Decision frequency | Frequent (designing systems) | Rare (exceptions only) |
| Task types | EXPLORE + EXPLOIT | Mostly EXPLOIT + ADMIN |
| Strategy changes | Common (discovering what works) | Infrequent (system proven) |
| Automation level | Low (manual with logging) | High (L6+ maturity) |
| Morning briefing | Heavy context, many decisions | Lightweight, mostly confirmations |
| Review focus | "Is this working?" | "Is anything broken?" |

## 10. Process for Adding to the System

### Add a New Goal

1. Copy `~/Master/task-system/goal-template-v1.3.md` → `~/Master/goals/GOAL-XXX.md`
2. Fill header: name, metric, measurement, deadline, status=ACTIVE, mode=BUILDING
3. Assess all 8 foundations
4. Define sub-goals, mechanism chains, paths
5. Set MVP path, threshold gates, system stop criteria
6. Create tasks for MVP path foundations
7. Link `Strategy Change Protocol: strategy-change-protocol.md`

### Add a New Path

1. In goal file Paths table: add row with ID, name, which step it feeds, status
2. If not MVP: consider deferring behind a threshold gate
3. Create tasks needed for this path
4. Update task `Goal:` fields to reference new PATH-ID
5. Log decision in goal file Decision Log (Type 2 — reversible)

### Add a New Task

1. Follow AGENTS.md Task Creation Protocol (Steps 1-6)
2. Check registry → copy template → determine level → fill fields → score → classify owner
3. Set `Goal:` field linking to goal/path/mechanism step
4. Add to TASK-REGISTRY.md + mirror
5. Push to Master repo

### Add a New Persona

1. Copy Atlas section structure from `~/Master/personas/atlas-v1.md`
2. Replace: name, role, expertise, techniques, skill files, invocation method
3. Required sections: Role, When to invoke, Core Identity, Founding Principles, Core Expertise, Techniques, Anti-Patterns, Skill Files, How to Invoke
4. Add to personas directory

### Add a New Foundation

Foundations are fixed at 8 (from BMC/Lean Canvas synthesis). If you believe a 9th is needed:
1. Verify it's truly irreducible — can't be modelled as a sub-component of an existing foundation
2. Add to goal-template-v1.3.md foundation table
3. Add to all existing goal files
4. Update v1.3-lexicon.md with definition
5. Update neo4j-schema-design.md
6. Log decision with rationale

### Add a New Operational Process (Checklist, Cron, Review)

1. Determine hierarchy level: Global → Domain → Subdomain → Process
2. Create checklist file in `task-system/checklists/` following README.md architecture
3. Reference from task-field-guide.md domain checklists section
4. If cron-based: create cron job with appropriate schedule
5. If review-based: add to relevant cadence (daily/weekly/monthly/quarterly)

## Appendix A: EBAM Categories Quick Reference

| Cat | Name | Driver | Risk | Intervention Theme |
|-----|------|--------|------|-------------------|
| 🟢 E | Epistemic | High novelty, SEEKING system | Over-exploring, never finishing | Time-box, force output |
| 🟡 B | Boring | Low entropy, routine | Procrastination, rushing | Batch, automate, gamify |
| 🔴 A-S | Aversive-Social | Interpersonal risk, FEAR/PANIC | Avoidance, over-preparation | Reduce exposure, script, exposure ladder |
| 🔴 A-I | Aversive-Internal | Uncertain return, identity-relevant | Paralysis, perfectionism | Reframe, micro-commit, accountability |
| 🟠 M | Mixed | Multiple competing drivers | Competing interventions | Address dominant driver first |

## Appendix B: Dependency Types Quick Reference

| Type | Symbol | Meaning | Scanner Behaviour |
|------|--------|---------|-------------------|
| Hard | → | Can't proceed without | Blocks task |

## 11. Automation Enforcement

### 11.1 Cron Classification

Every cron in the system has a **Scope** — this determines whether it's infrastructure (required for the task system to function at all) or domain-specific (required for a specific business area to function).

| Scope | Meaning | Lifecycle | Example |
|-------|---------|-----------|---------|
| **🔧 System** | Global task manager infrastructure. Without it, the entire system breaks or degrades regardless of what domain you're working in. Exists because the task manager exists. | Permanent — exists as long as the task system exists | Integrity scan, completion audit, system health |
| **🏢 Domain** | Serves a specific business function. Without it, that domain breaks but the task system itself keeps working. Created when a domain is added, deleted when a domain is removed. | Tied to domain lifecycle — add/remove with domains | Expandi health, H1 sync, outreach block |

**The test:** If you deleted all business tasks and goals, would this cron still need to exist? If yes → 🔧 System. If no → 🏢 Domain.

### 11.2 Model Tiers

Every cron (and sub-agent task) must specify which model tier it runs on. This prevents credit blowout from running expensive models on cheap tasks.

| Tier | Model | Cost | Use For |
|------|-------|------|---------|
| **Strategy** | Opus 4.6 | $$$ | Interactive strategy work ONLY. Never crons. Goal creation, system design, complex writing. |
| **Judgment** | Sonnet | $$ | Crons requiring context + decisions: morning briefing, evening block, weekly review prep. |
| **Validation** | Flash | $ (near-free) | Crons doing checks, syncs, scans: integrity, health, sync, threshold, monthly/quarterly triggers. |

**Rule:** No cron runs on Opus. If a cron needs Opus-level reasoning, it's not a cron — it's a task that should be scheduled as a sub-agent spawn with explicit model override.

### 11.4 The Meta-Rule

SYSTEM-HEALTH monitors all other crons. If it fails, the entire automation layer is blind. This is the ONE cron that must be manually verified during the weekly review (§7.2).

### 11.5 Adding Domain Crons

When a new business domain is added (e.g., content marketing, finance tracking):
1. Identify what recurring checks that domain needs
2. Create crons with Scope: 🏢 Domain and appropriate model tier
3. Add to `v1.3-cron-registry.md` under a new domain section
4. Reference from the domain's operational checklist

When a domain is removed:
1. Delete its domain crons
2. Remove from cron registry
3. System crons continue unaffected
