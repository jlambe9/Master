## Executive Summary

ATLAS-1 built a textbook-quality systems architecture. That's the problem. Jamie doesn't need textbook-quality — he needs **minimum viable structure** that he'll actually use with a 2nd-percentile Glr and ADHD executive function challenges.

The core insight (mechanism chain + bottleneck identification + shared requirements) is excellent and worth keeping. But the draft wraps it in enough ceremony to make it feel like enterprise project management software. For someone who struggles with state-tracking across hierarchies, this framework has **too many entity types, too many tables to maintain, and too many fields that will go stale**.

## Section-by-Section Critique

### §1. The Hierarchy — One Layer Too Many

**The claim:** Five entity types across two layers.
**The problem:** That's five things to track mentally. Jamie's working memory budget is already taxed.

```
Goal → Sub-Goal → Mechanism Step → Path → Task → Subtask
```

That's **SIX levels** if you count subtasks. For a guy with 2nd-percentile Glr, this is a retrieval nightmare. He'll forget where he is in the hierarchy and lose the "why am I doing this?" thread — the exact problem the framework is supposed to solve.

**Sub-Goals don't earn their keep.** In the worked example, SG-001 ("Sign paying coaching clients") is basically just the goal restated at slightly lower altitude. SG-002 is blocked and empty. For Jamie's current situation (one goal, one real funnel), sub-goals add a level of indirection that creates cognitive cost without value.

**Recommendation:** Collapse Sub-Goal into Goal. A goal HAS a mechanism chain directly. If Jamie ever has a goal complex enough to need sub-goals, add the layer then. YAGNI.

This reduces the hierarchy to:
```
Goal → Mechanism Step → Path → Task
```
Four levels. Holdable in working memory. Three hops from task to goal.

### §2. Goal File Template — Too Many Tables

Count the tables in the template: **7 tables + 1 checklist**. That's 7 things Jamie has to scan and potentially update.

| Table | Will Jamie update this? | Verdict |
|-------|------------------------|---------|
| Sub-Goals | Already argued to remove | ❌ CUT |
| Mechanism Chain | Yes — this is the core value | ✅ KEEP |
| Paths | Yes, occasionally | ✅ KEEP |
| Shared Requirements | Yes — high value, shows blockers | ✅ KEEP |
| Threshold Gates | Maybe once a month? | ⚠️ DEFER |
| System Stop Criteria | Will be filled once, never updated | ❌ CUT |
| Decision Log | Has never been used in any task file to date | ❌ CUT (or make optional) |

**Threshold Gates** are a nice-to-have but they're speculative. Jamie has zero clients. Gates about "≥3 clients" are premature. This is the kind of thing you add when you're approaching a gate, not when you're at zero. Right now it's architecture astronautics — building toll booths on a road nobody's driving on yet.

**System Stop Criteria** is a clever concept that will be written once and ignored. It's optimising for a problem Jamie doesn't have yet (over-building the system). The real protection against over-building is the bottleneck analysis itself — if you're always working on the bottleneck, you're not over-building by definition.

**Decision Log** has been in task templates since v1.2. I'd bet money it's empty in every single task file. If nobody's using it at task level, nobody's using it at goal level. Don't cargo-cult it upward.

**Recommendation:** The goal file should have **3 tables**: Mechanism Chain, Paths, Shared Requirements. That's it for v1. Everything else is added when there's evidence it's needed.

### §3. Relationship Model — Architecture Astronautics

This section defines 12 relationship types across 8 entity types. This is a schema for a team of engineers, not a solo ADHD founder with an AI assistant.

**"Shared Requirements are a different entity type, implemented by tasks" (Q3)** — This is the single most over-engineered decision in the document. A shared requirement IS a task that blocks multiple things. Making it a separate entity means:
- A new concept to learn
- A new table to maintain
- A mapping layer (SR → implementing tasks) that duplicates information
- Status derivation logic ("SR status = f(implementing task statuses)")

ATLAS-1 even flagged the simpler alternative: "just tag certain tasks as `shared: true`." That's the right answer. A task with `blocks: [M2, M3]` and `shared: true` gives you everything the SR entity gives you, without a new concept.

**Recommendation:** Kill the SharedRequirement entity. Use tagged tasks. The "clean goal-level view" ATLAS-1 wants can be generated by querying tasks where `shared: true`.

### §4. Neo4J Schema — Premature

The entire Neo4J section is premature optimization. There is no Neo4J database. There is no sync script. There are no queries to run. This section is designing a database for data that doesn't exist yet.

**Would Jamie actually use this?** No. Jamie would read his goal markdown file. Rich (the agent) might query Neo4J eventually, but that's a v1.5 concern.

**The risk:** By designing the schema now, the framework couples itself to a specific implementation. If the markdown structure changes (which it will, because this is v1.3 draft), the schema has to change too. Two things to update instead of one.

**Recommendation:** Delete §4 entirely from the goal framework. Put it in a separate `neo4j-schema-design.md` when there's an actual graph database to populate. The framework should be markdown-native and work without Neo4J.

### §5. Worked Example — Actually Good

This is the strongest section. It makes the framework concrete and immediately reveals actionable insights (shared requirements are the real blocker, not pipeline). The "What This Reveals" callout is exactly the kind of thing that proves the framework's value.

**One issue:** The example inherits all the over-engineering from the template (threshold gates at zero, system stop criteria, empty decision log). A leaner example would be more honest about what Jamie actually needs right now.

### §6. Critical Questions — Mixed

**Q1 (inputs/outputs at different levels):** Good analysis, right conclusion.

**Q2 (shared requirement status):** Good concept, over-engineered implementation (see §3 critique).

**Q3 (SR entity type):** Wrong decision. The "simpler alternative" ATLAS-1 dismissed is the better answer.

**Q5 (avoiding over-engineering):** Ironic section. The three rules are excellent — "the goal framework is a LENS on existing tasks, not a new dependency layer." But the framework itself violates this principle by introducing 3 new entity types (SharedRequirement, ThresholdGate, MechanismStep as a formal node) instead of being a lens.

**Q6 (Neo4J translation):** Premature (see §4).

### §7. Uncertainties — Reveals the Problem

The "Needs Jamie's Decision" list has 4 items. The "Known Gaps" list has 4 items. That's 8 open questions on a framework that hasn't been used once. This is a sign that the framework is being designed in the abstract rather than emerging from use.

**"Should Engage split into two steps?"** — You have zero data points. This question can't be answered. Don't even ask it yet.

**"Where do metrics live?"** — In the markdown file, updated by hand, until there's enough data to justify automation. Deciding on Sheets vs Neo4J vs dedicated file before you have a single metric reading is premature.

**"Path-level conversion tracking"** — "only after ≥20 data points per step" — correct instinct, but then why design it now?

## Theory of Constraints Analysis

**What is the MINIMUM structure for the goal layer to be useful?**

The goal layer needs to answer exactly two questions:
1. **Why am I doing this task?** (trace from task → goal)
2. **What should I work on next?** (bottleneck identification)

To answer those, you need:
- A goal (one-liner + metric + deadline)
- A mechanism chain (the funnel with conversion ratios)
- Paths (what feeds the funnel) linked to tasks
- Knowledge of what's blocking the funnel (shared blockers)

You do NOT need:
- Sub-goals (one goal, one funnel — add sub-goals when there are multiple funnels)
- Threshold gates (no data to gate on)
- System stop criteria (solved by bottleneck focus)
- Decision log (not used in practice)
- Neo4J schema (no database exists)
- Formal SharedRequirement entity (tagged tasks suffice)
- 12 relationship types (4 suffice: Goal→MechanismStep, MechanismStep→MechanismStep, Path→MechanismStep, Path→Task)

## The Minimum Viable Goal Framework

If I were designing this for Jamie's brain, it would be:

```markdown
# GOAL-001 — £5k/month Recurring Profit

**Target:** £5,000/month net profit by 2026-06-30
**Status:** ACTIVE

## The Funnel

| Step | In → Out | Ratio | Now | Bottleneck? |
|------|----------|-------|-----|-------------|
| Reach | Contacts → Conversations | 5% | 20/wk | |
| Engage | Conversations → Calls | 10% | 0/wk | ⚠️ |
| Close | Calls → Clients | 20% | 0/wk | |

**Bottleneck:** Engage — no calls booked. Missing: offer, booking link.

## What Feeds the Funnel

| Path | Status | Tasks |
|------|--------|-------|
| LinkedIn | ACTIVE (MVP) | LP-001→006 |
| Meetups | NOT STARTED | — |

## Blockers (shared across all paths)

| What's Missing | Task | Blocks |
|----------------|------|--------|
| Coaching offer | BC-OFFER-V1 | Engage, Close |
| Booking link | TBD | Engage |
| Payment system | TBD | Close |
| Legal | BOSS-GDPR | Close |
```

That's it. Three tables. One page. Holdable in working memory. Answers both core questions (why am I doing this? what's next?). Everything else is earned complexity — add it when there's evidence it's needed.

## Summary of Recommendations

| Item | Action | Rationale |
|------|--------|-----------|
| Sub-Goals | **CUT** | One goal = one funnel. Add when multiple funnels exist. |
| Threshold Gates | **DEFER** | Zero data to gate on. Add when approaching a gate. |
| System Stop Criteria | **CUT** | Bottleneck focus prevents over-building. This won't be maintained. |
| Decision Log | **CUT** (or optional) | Not used in practice at task level. Won't be used here. |
| SharedRequirement entity | **CUT** | Use tagged tasks with `shared: true` instead. |
| Neo4J Schema (§4) | **MOVE** to separate doc | Premature. Framework should be markdown-native. |
| Relationship Model (§3) | **SIMPLIFY** to 4 relationships | 12 is architecture astronautics for a solo founder. |
| Goal file tables | **REDUCE** from 7 to 3 | Funnel, Paths, Blockers. That's the minimum viable goal file. |
| Worked example | **KEEP** but make leaner | Best section. Trim to match simplified template. |
| Critical Questions (§6) | **KEEP Q1, Q5** | Good principles. Q3 wrong answer. Q6 premature. |

**Net effect:** Framework goes from ~8 entity types and 7 tables to ~4 entity types and 3 tables. Cognitive load roughly halved. Everything that was cut can be added back when evidence justifies it.
