# v1.3 Goal Framework — Knowledge Graph Integrity Critique

> **Reviewer:** ATLAS-3 (Knowledge Graph Integrity)
> **Date:** 2026-02-22
> **Reviewing:** v1.3-goal-framework-draft.md
> **Verdict:** STRONG FOUNDATION, 7 ISSUES TO RESOLVE BEFORE IMPLEMENTATION

---

## Summary

The schema is well-designed for a first pass. The entity separation is clean, the key queries work, and the strategy/execution layering is sound. But there are real problems that will bite during implementation — ambiguous edges, a phantom node type, a sync design that doesn't exist yet, and versioning that's completely unaddressed.

---

## Issue 1: SharedRequirement Has a Redundant `blocksStep` Property AND a `BLOCKS` Edge

**Severity: 🔴 HIGH — data inconsistency guaranteed**

The `SharedRequirement` node has:
- A `blocksStep: String` **property** ("Which mechanism step(s) this blocks")
- A `(:SharedRequirement)-[:BLOCKS]->(:MechanismStep)` **edge**

These WILL desync. The property is a denormalized copy of the edge. In a graph database, the edge IS the canonical representation. The property must go.

Additionally, `blocksStep` is typed as `String` but the worked example shows a SR blocking multiple steps ("M2, M3"). A string with comma-separated values is an anti-pattern in a graph — that's what multiple edges are for.

**Fix:** Remove `blocksStep` property from `SharedRequirement`. The `BLOCKS` edges are the single source of truth. If you need the info in markdown, derive it from the graph.

---

## Issue 2: `MetricSource` Node Appears in Edges but Not in Node Types

**Severity: 🔴 HIGH — orphaned reference**

§4 Edge Types includes:
```
(:ThresholdGate)-[:READS_FROM {field: String}]->(:MetricSource)
```

But `MetricSource` is never defined in the Node Types section. What is it? A Stripe dashboard? An H1 sheet? A Neo4J property on another node? This is a phantom node type — any query involving it will fail.

**Fix:** Either:
- (a) Define `(:MetricSource {id, name, type, url, refreshFrequency})` as a proper node type, OR
- (b) Collapse it — make `source` a property on `ThresholdGate` (simpler, appropriate if metric sources aren't shared across gates). Given the current scale (~3 gates per goal), option (b) is sufficient. Promote to a node when you have >10 gates sharing sources.

---

## Issue 3: `CHAINS_TO` vs `HAS_MECHANISM` — Ambiguous Ownership of Order

**Severity: 🟡 MEDIUM — query confusion**

Mechanism steps have ordering expressed in TWO places:
1. `(:SubGoal)-[:HAS_MECHANISM {order: Int}]->(:MechanismStep)` — order property on the ownership edge
2. `(:MechanismStep)-[:CHAINS_TO {order: Int}]->(:MechanismStep)` — explicit chain edges between steps

These are redundant. `CHAINS_TO` already implies order (it's a linked list). Adding `order` to `HAS_MECHANISM` creates a second source of truth for sequence. If someone reorders `CHAINS_TO` but forgets `HAS_MECHANISM.order`, queries return contradictory results.

**Fix:** Pick one:
- **Option A (recommended):** Keep `CHAINS_TO` for ordering. Remove `order` from `HAS_MECHANISM`. The chain IS the order. Use `HAS_MECHANISM` only for ownership (which SubGoal owns this step).
- **Option B:** Remove `CHAINS_TO` entirely. Use `HAS_MECHANISM.order` for sequence. Simpler, but loses the explicit pipeline topology and makes "what comes after step X?" a sort query instead of a traversal.

I recommend Option A. `CHAINS_TO` is more expressive — it supports branching funnels later (e.g., M2 splits into M2a and M2b) without schema changes.

## Issue 4: No Versioning Model — "If a Mechanism Chain Changes, What Breaks?"

**Severity: 🔴 HIGH — the draft explicitly punts on this**

ATLAS-1's Principle 2 is "Temporal Process Modelling" — processes evolve, history preserved. But the draft has zero versioning:

- If M2 (Engage) splits into M2a (Start Conversation) and M2b (Book Call), what happens to:
  - Historical throughput data on the old M2?
  - Paths that `FEEDS` the old M2?
  - SharedRequirements that `BLOCKS` the old M2?
  - ThresholdGates that `READS_FROM` metrics tied to M2?
  - The execution log entries that reference M2?

- If a new Path is added and an old one dropped, the historical conversion data is lost unless versioned.

- Goal file `Framework: v1.3` exists but there's no mechanism for what happens when v1.4 changes the schema.

**Fix:** Add a `version` property to Goal nodes and a `(:Goal)-[:HAS_VERSION {version: String, date: Date}]->(:GoalSnapshot)` pattern. OR (simpler for now): treat mechanism changes as Decision Log entries in the goal file, and add `deprecated: Boolean, replacedBy: String` properties to MechanismStep. Don't over-engineer temporal modelling until the second goal exists — but acknowledge the pattern now so the schema doesn't prevent it later.

## Issue 5: Task ↔ Goal Bidirectional Reference — The Sync Nightmare

**Severity: 🔴 HIGH — this is the hardest practical problem**

The draft says tasks are "existing v1.2 — no changes." But for the graph to work, tasks need to know which Path they belong to. Currently, task files have:

```markdown
**Domain:** boss/lead-gen
**Dependencies:** Hard → [task IDs]
```

There's no `Path:` or `Goal:` field. The graph relationship `(:Path)-[:CONTAINS_TASK]->(:Task)` exists only in Neo4J (or the goal file's Paths table). This means:

1. **Goal file is the authoritative source for Path→Task mapping.** If someone adds a task to a path in the goal file but doesn't update the task file, the task file doesn't know it serves a goal.
2. **Task files can't trace upward** without reading the goal file. The "why am I doing this?" query requires parsing goal files.
3. **If a task is shared across paths** (not addressed — can it be?), you need multiple `CONTAINS_TASK` edges, but the goal file table syntax doesn't support this cleanly.

**Fix:** Add an optional field to the v1.2 task template:

```markdown
**Goal:** GOAL-001 | Path: PATH-LI | Step: M1
```

This is a soft reference (not enforced), but it enables:
- Bottom-up tracing from task files without loading goal files
- The sync script to validate bidirectional consistency
- Humans reading a task file to understand strategic context

**Sync rule:** Goal file is authoritative for Path→Task membership. Task file's `Goal:` field is a convenience mirror. The sync script flags mismatches.

## Issue 6: `UNLOCKS` Edge is Polymorphic — Target Can Be Task, Path, or SubGoal

**Severity: 🟡 MEDIUM — query complexity**

```
(:ThresholdGate)-[:UNLOCKS]->(:Task | :Path | :SubGoal)
```

In Neo4J this works fine (edges can connect any node types), but queries become awkward:

```cypher
MATCH (tg:ThresholdGate)-[:UNLOCKS]->(target)
// What type is target? Need to check labels.
```

This isn't a dealbreaker, but consider: do you actually need gates to unlock SubGoals? The worked example only shows gates unlocking Tasks and Paths. Unlocking a SubGoal seems like it should be a new Goal, not a gated SubGoal.

**Fix:** If SubGoal unlocking is real, keep it but document it. If not, constrain `UNLOCKS` to `Task | Path` only. Either way, add a `targetType` property on the `UNLOCKS` edge for efficient filtering:

```cypher
(:ThresholdGate)-[:UNLOCKS {targetType: 'Path'}]->(:Path)
```

## Issue 7: Sub-Goal Could Be a Property, Not a Node (At Current Scale)

**Severity: 🟢 LOW — worth noting, not blocking**

With 1 goal and 2 sub-goals, `SubGoal` as a separate node type adds schema complexity for minimal traversal benefit. The only query that traverses through SubGoal is the task→goal trace, and it adds a hop.

However: the draft correctly identifies that different sub-goals may have different mechanism chains. This justifies the node. If every sub-goal shared the same mechanism chain, I'd argue for collapsing it.

**Verdict:** Keep as node. But re-evaluate if goals consistently have only 1 sub-goal — that's a sign the layer is unnecessary.

## Hop Count Analysis

The draft claims ≤4 hops from any task to its goal. Let me verify:

| Query | Path | Hops | ✅/❌ |
|-------|------|------|-------|
| Task → Goal | Task → Path → MechanismStep → SubGoal → Goal | 4 | ✅ |
| SharedReq → Goal | SharedReq ← REQUIRES ← Goal | 1 | ✅ |
| "What blocks Goal X?" | Goal → SharedReq (status=INCOMPLETE) | 1 | ✅ |
| "What's the bottleneck?" | Goal → SubGoal → MechanismStep (isBottleneck) | 2 | ✅ |
| "What feeds the bottleneck?" | + → Path (via FEEDS) | 3 | ✅ |
| "What tasks feed the bottleneck?" | + → Task (via CONTAINS_TASK) | 4 | ✅ |
| "What shared reqs block step X?" | MechanismStep ← BLOCKS ← SharedReq | 1 | ✅ |
| "Fastest path to goal?" | Goal → SubGoal → MechStep → Path (sort by conversion) | 3 | ✅ |
| ThresholdGate status | Goal → ThresholdGate | 1 | ✅ |

**All key queries ≤4 hops. No circular dependencies detected.** The graph is a clean DAG with the exception of `CHAINS_TO` (which is a linear chain within a sub-goal, not circular).

## Circular Dependency Check

No cycles exist in the schema as designed. However, there's a latent risk:

- `Task DEPENDS_ON Task` (existing v1.2) can create cycles if not validated. This is a pre-existing issue, not introduced by v1.3.
- `SharedRequirement IMPLEMENTED_BY Task` + `Task` in a `Path` that `FEEDS` a step `BLOCKED` by that same SR — this isn't circular (SR blocks the step, not the path), but it could confuse humans reading the graph. Document that "BLOCKS means the step can't fully convert, not that the path can't operate."

## Missing: Key Query Not Supported — "What Shared Requirements Are Missing?"

The draft's Query 1 handles this for a single goal:
```cypher
MATCH (g:Goal {id:'GOAL-001'})-[:REQUIRES]->(sr {status:'INCOMPLETE'})
```

But "shared requirements" across goals (if multiple goals exist) needs:
```cypher
MATCH (sr:SharedRequirement {status:'INCOMPLETE'})<-[:REQUIRES]-(g:Goal)
RETURN sr.name, collect(g.name) as affectedGoals
```

This works but only because SRs are goal-scoped. If two goals share the same requirement (e.g., "Stripe payment system"), should there be ONE SR node referenced by both goals, or two separate SR nodes? The draft doesn't address multi-goal SR sharing.

**Fix:** Add a uniqueness constraint: SRs are identified by name/id globally. If two goals need Stripe, they both `REQUIRES` the same SR node. This prevents duplicate work tracking.

## Integration with Existing Task System (v1.2)

### What Works
- Task files remain unchanged — no migration needed
- `CONTAINS_TASK` edges are additive (goal files reference tasks, tasks don't need to change)
- Registry stays as-is — goals are a layer above, not a replacement

### What Needs Work
1. **Task template needs optional `Goal:` field** (see Issue 5)
2. **TASK-REGISTRY.md needs a `Goal` column** — so you can see at a glance which tasks serve which goals. Currently the registry groups by domain (boss/lead-gen), not by goal.
3. **Sync script specification is missing** — how do markdown goal files become Neo4J nodes? The draft mentions "a sync script reads goal files → upserts Neo4J nodes/edges" but doesn't spec it. This is the most important piece of infrastructure for graph integrity.
4. **SUBSUMED tasks** — 6 tasks in the registry are SUBSUMED into LP series. In the goal framework, these don't appear. But their Neo4J nodes would still exist. Need a `SUBSUMED_BY` edge or a cleanup policy.

## Final Assessment

The schema is **graph-ready with corrections.** The entity model is sound — five node types in two layers is the right level of abstraction. The key queries all work within acceptable hop counts. No circular dependencies.

The critical gaps are operational, not conceptual: versioning, sync, and the redundant `blocksStep` property. Fix the 🔴 issues before writing the sync script, and this will translate cleanly to Neo4J.
