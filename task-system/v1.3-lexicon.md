# v1.3 Lexicon — Precise Terminology for the Goal/Task System

> **Author:** ATLAS v1
> **Date:** 2026-02-22
> **Status:** DRAFT
> **Purpose:** Single source of truth for every term used in v1.3. Eliminates ambiguity. Each term: name, definition, example, source discipline.

---

## Strategic Layer

### Goal
**Definition:** A measurable business outcome with a deadline. One metric, one target value, one date. Goals are stable — changing a goal is a strategic pivot, not a task update.
**Example:** "£5k/month recurring profit by 2026-06-30" (GOAL-001).
**Source:** MBO (Management by Objectives, Drucker), OKR (Objectives & Key Results, Grove/Doerr), Hoshin Kanri.
**System rule:** Goals live in `~/Master/goals/`. A goal file contains sub-goals, mechanism chains, paths, shared requirements, threshold gates, and system stop criteria.

### Sub-Goal
**Definition:** A decomposed target that, combined with siblings, achieves the parent goal. Still measurable with its own metric and target. Sub-goals partition the goal into independently trackable streams.
**Example:** "Sign 6 paying clients @ £500+/month" (SG-001 under GOAL-001).
**Source:** WBS (Work Breakdown Structure), goal decomposition (systems engineering).
**System rule:** Sub-goals are sections within the goal file, not separate files (unless >3 sub-goals). Each sub-goal has its own mechanism chain.
**Note:** We keep sub-goals. They serve a distinct function — a goal may have revenue AND expense sub-goals with completely different mechanism chains.

### Mechanism Step
**Definition:** One stage in a conversion funnel that transforms an input quantity into an output quantity at a measurable ratio. Mechanism steps chain sequentially. The step with the worst ratio or lowest throughput is the **bottleneck**.
**Example:** "M1: Reach — People contacted → Conversations started @ 5%."
**Source:** Theory of Constraints (Goldratt), Value Stream Mapping (Lean), Sales/Marketing funnel theory.
**System rule:** Mechanism steps are continuous processes, not tasks that "complete." They have throughput, not status. They chain: M1 → M2 → M3.

### Mechanism Chain
**Definition:** The ordered sequence of mechanism steps from initial input to final output for a sub-goal. The chain IS the conversion funnel. Reading the chain reveals where value is created and where it leaks.
**Example:** "Reach → Engage → Close" for SG-001.
**Source:** Value stream mapping (Lean), sales pipeline theory, Theory of Constraints.
**System rule:** Every sub-goal has exactly one mechanism chain.

### Path
**Definition:** An interchangeable channel, method, or tactic that feeds a mechanism step with throughput. Paths are where execution lives. Multiple paths can feed the same step. Paths can be added, paused, or dropped without changing strategy.
**Example:** "LinkedIn Outreach" (PATH-LI) feeds M1: Reach.
**Source:** Channel strategy (marketing), portfolio theory (multiple bets on the same outcome).
**System rule:** Paths are the bridge between strategy and execution. Above = stable strategy. Below = volatile tasks. A path contains tasks.

### Business Primitive
**Definition:** An irreducible infrastructure component that the business cannot function without. Primitives are not tasks — they are *conditions* that must be true. They cut across all paths and mechanism steps.
**Example:** "Payment system exists" — without this, no path can close a sale regardless of how good the funnel is.
**Source:** Business Model Canvas (Osterwalder), infrastructure architecture, platform thinking.
**System rule:** In v1.3, these are modelled as **Shared Requirements** — goal-level conditions implemented by tasks that block mechanism steps.

### Value Stream
**Definition:** The end-to-end flow of activities that deliver value to a customer, from first contact to payment received. In Jamie's system, the value stream IS the mechanism chain plus the paths feeding it.
**Example:** Contact on LinkedIn → Conversation → Discovery call → Sign contract → Receive payment.
**Source:** Lean manufacturing (Womack & Jones), Value Stream Mapping.
**System rule:** The mechanism chain models the quantitative side of the value stream. The paths model the qualitative/tactical side.

### Constraint (Theory of Constraints)
**Definition:** The single factor that limits the system's throughput toward its goal. At any time, there is exactly ONE constraint (bottleneck). Improving anything other than the constraint is waste.
**Example:** "M2: Engage is the bottleneck — generating reach but zero calls booked, because shared requirements (offer, booking link) aren't built."
**Source:** Theory of Constraints (Goldratt, *The Goal*, 1984).
**System rule:** The goal file has a "Current Bottleneck" field. The bottleneck determines where effort goes. When the bottleneck moves (because you fixed it), the system recalibrates.

### Bottleneck
**Definition:** The mechanism step currently operating as the constraint. Identified by the lowest throughput or worst conversion ratio in the chain. Synonym for "constraint" in this system — used interchangeably.
**Example:** M2: Engage — 0 calls booked/week.
**Source:** Theory of Constraints, queueing theory.

### Leading Indicator
**Definition:** A metric that predicts future outcomes. It measures input activity or early-stage results that will *eventually* produce the lagging indicator. Controllable, forward-looking.
**Example:** "Connection requests sent per week" — predicts future conversations. "Conversations started" — predicts future calls.
**Source:** Balanced Scorecard (Kaplan & Norton), process control theory.
**System rule:** Mechanism step inputs and early-stage outputs are leading indicators. Track these for operational control.

### Lagging Indicator
**Definition:** A metric that measures outcomes after the fact. It confirms whether the strategy worked but cannot be directly controlled. Backward-looking.
**Example:** "Revenue this month," "Clients signed this quarter."
**Source:** Balanced Scorecard, financial reporting.
**System rule:** Goal metrics and sub-goal metrics are typically lagging indicators. Don't manage daily by lagging indicators — use leading indicators for that.

### Strategy Deployment (Hoshin Kanri)
**Definition:** The practice of translating strategic goals into aligned execution at every level, with regular review cadences to ensure alignment is maintained. "Direction management" — everyone rows the same way.
**Example:** GOAL-001 → SG-001 → Mechanism Chain → PATH-LI → LP-001…LP-006. Every task traces to the goal.
**Source:** Hoshin Kanri (Japanese strategic planning, Toyota), catchball process.
**System rule:** The v1.3 framework IS strategy deployment. The trace from task → goal (≤4 hops) is the alignment mechanism.

### Run Rate / Maintenance Mode
**Definition:** The steady-state operation of a system that has been built and is now self-sustaining. Characterized by: low time investment (<2hrs/day), predictable outputs, monitoring replaces active management, interventions are exception-based.
**Example:** "LinkedIn outreach pipeline is in run mode — Expandi sends CRs, intros are semi-automated, screening runs on schedule."
**Source:** Operations management, DevOps (run books), ITIL (service operation).
**System rule:** System Status field: BUILDING → OPERATIONAL → OPTIMISING. "OPERATIONAL" = run mode. The System Stop Criteria checklist defines when to transition.

### Build Mode / Project Mode
**Definition:** The active construction phase of a system or process. Characterized by: high time investment, frequent decisions, creating new infrastructure, defining processes. Build mode should have a defined end point (the system stop criteria).
**Example:** "Currently building the coaching business pipeline — creating offer, booking system, payment flow."
**Source:** Project management (PMBOK), build vs run distinction (IT operations).
**System rule:** System Status = BUILDING. Build mode tasks are finite. If build mode never ends, scope is wrong.

### Stop Criteria
**Definition:** A checklist of conditions that, when all are met, mean the *system* (not the goal) is considered "done building." Prevents endless infrastructure work. After stop criteria are met, the system transitions to run mode.
**Example:** "All paths mapped, MVP path operational, shared requirements complete, automatable tasks automated, <2hrs/day steady state."
**Source:** Operations readiness review (NASA, military), definition of done (Scrum), go/no-go criteria.
**System rule:** Every goal file has a System Stop Criteria section. These are checked during strategy reviews.

### Operational Readiness
**Definition:** The state where all prerequisites for a system to function are verified and active. More granular than stop criteria — it's the pre-flight check that confirms everything is actually working *right now*.
**Example:** Domain operational checklists: "Expandi session connected? H1 sheet accessible? API keys valid?"
**Source:** Operations readiness review, pre-flight checklists (aviation), runbook verification.
**System rule:** Domain checklists in `task-system/checklists/` verify operational readiness. Checked during dependency tracing — if a checklist item fails, that's the root blocker.

### Threshold Gate
**Definition:** A quantitative condition that, when met, unlocks a new action, path, or investment. Prevents premature building. The gate has a metric, a threshold value, a current value, and a source.
**Example:** "TG-001: Don't build group program until ≥3 clients signed. Current: 0. Status: LOCKED."
**Source:** Stage-gate process (Cooper), evidence-based management, lean startup (validated learning).
**System rule:** First-class entities in the goal file. Scanner monitors current values. At 80%: "start preparing." At 100%: unlock and alert.

### Deferred Work
**Definition:** Work that is intentionally not started because a threshold gate is locked or because it's below the current bottleneck in priority. Not "forgotten" — actively parked with a trigger condition for when it becomes relevant.
**Example:** "Build onboarding process (BOSS-ONBOARD) — deferred until TG-003 (≥1 client signed) unlocks."
**Source:** Lean (just-in-time), real options theory, YAGNI (extreme programming).
**System rule:** Deferred work is linked to threshold gates. When the gate unlocks, the work transitions from deferred to CAPTURED.

## Execution Layer

### Task
**Definition:** A discrete unit of work with a defined output, owner, and status. Tasks live inside paths and implement the work needed to feed mechanism steps. Tasks have completion gates — they are done when all outputs verifiably exist.
**Example:** "LP-003 — Write Personalised Intros" — a task within PATH-LI.
**Source:** WBS (Work Breakdown Structure), GTD (Getting Things Done, Allen), Kanban.
**System rule:** v1.2 task files. No changes to task structure in v1.3. Tasks gain context (which path → which mechanism step → which goal) but not new fields.

### Subtask
**Definition:** An atomic unit of work within a task. One subtask = one output. Subtasks have their own owner (which may differ from the parent task's owner) and their own completion gate.
**Example:** "Subtask 3: Write intro message — Owner: 🤖 Rich — Output: intro text in H1 column."
**Source:** WBS decomposition, Kanban.
**System rule:** Table format: `| # | Type | Task | Status | Owner | Output (proof of completion) |`. Parent is complete only when ALL subtasks are complete AND parent win state outputs verified.

### Output
**Definition:** A specific, verifiable artifact or state change that proves work is done. Not "worked on" or "attempted" — a concrete thing that exists and can be checked. Every task and subtask must have defined outputs.
**Example:** "Intro text written in H1 column J for each lead" — can be verified by reading the column.
**Source:** Definition of Done (Scrum), output-based management, evidence-based completion.
**System rule:** The completion gate checks outputs. No output defined = task can't be marked done = task spec is incomplete.

### Completion Gate
**Definition:** The verification checkpoint before marking a task/subtask done. ALL outputs in the win state must verifiably exist. Owner-specific rules apply (Rich verifies; Hybrid needs Jamie's approval for human outputs; Jamie-only tasks only Jamie marks done).
**Example:** "Before marking LP-003 done: check H1 column J is populated for all leads in batch."
**Source:** Quality gates (CMMI), definition of done (Scrum), acceptance criteria.
**System rule:** Full protocol in AGENTS.md → Step 5. Non-negotiable. Vague win states = can't complete = flag for definition.

### Owner Types

#### 🤖 Rich (Agent E2E)
**Definition:** Steps fully defined, inputs available, output machine-verifiable. No subjective quality gate. Agent executes autonomously.
**Example:** "Scrape LinkedIn profiles and populate H1 columns."

#### 🔄 Hybrid (Agent + Jamie)
**Definition:** Agent does some steps, Jamie reviews/approves/decides at checkpoints. Agent produces draft, Jamie makes final call.
**Example:** "Rich writes intros, Jamie validates tone before sending."

#### 🧑 Jamie (Human Only)
**Definition:** Physical task, personal presence required, task undefined (Jamie's job IS to define it), or output not verifiable by agent.
**Example:** "Have discovery call with prospect."

#### ❓ Undefined
**Definition:** Not enough information to determine owner. This IS a flag — task needs definition before it can progress.

**Source for all:** RACI matrix concepts, automation pyramid, human-in-the-loop patterns.
**System rule:** Owner progresses 🧑→🔄→🤖 as task matures. Criteria in task-field-guide.md.

### EBAM Categories
**Definition:** The four behavioural categories that determine how a task should be supported, based on Jamie's cognitive/affective profile:

- **🟢 E (Epistemic):** High novelty, high information entropy, aligns with SEEKING. Jamie naturally pursues these. Risk: over-exploring, never finishing. Intervention: time-box, force output.
- **🟡 B (Boring):** Low entropy, routine, specified procedure. Under-stimulates SEEKING. Risk: procrastination, rushing. Intervention: batch, automate, gamify, pair with E task.
- **🔴 A-S (Aversive-Social):** Interpersonal risk — rejection, judgment, exposure. Triggers FEAR/PANIC. Risk: avoidance, over-preparation. Intervention: reduce exposure, script interactions, exposure ladder.
- **🔴 A-I (Aversive-Internal):** Uncertain return, identity-relevant, confrontation. Triggers internal resistance. Risk: paralysis, perfectionism. Intervention: reframe, micro-commit, accountability.
- **🟠 M (Mixed):** Multiple drivers compete. Intervention must address the dominant driver without amplifying others.

**Source:** Jaak Panksepp's affective neuroscience (SEEKING, FEAR, PANIC systems), CHC cognitive theory, Layer 0 scoring.
**System rule:** Category is computed from Layer 0 scores × Jamie's profile. Determines which interventions are loaded.

### Dependencies

#### Hard Dependency (→)
**Definition:** Cannot physically proceed without the upstream task's output. The input literally doesn't exist.
**Example:** "Can't send intros without intros being written."

#### Soft Dependency (~>)
**Definition:** Upstream task improves effectiveness but isn't required. Can proceed without it, just not as well.
**Example:** "Offer doc makes sales calls better but conversations can happen without it."

#### Threshold Dependency (⊕)
**Definition:** Proceeds when a quantitative condition is met. Upstream doesn't "complete" — it produces accumulating signal. Three components: source task, metric + threshold, source location.
**Example:** "Don't build automation until process run 5+ times manually" (⊕ execution_count ≥ 5).

**Source for all:** Critical Path Method (CPM), dependency theory, project scheduling.
**System rule:** Hard deps block. Soft deps inform. Threshold deps gate. Scanner monitors thresholds at 80%/100%.

### Maturity Levels
**Definition:** A progression scale for task profiles, tracking how systematised and automated a repeatable process has become.

| Level | Name | Meaning |
|-------|------|---------|
| L0–L5 | Pre-RTP | Not all RTP fields filled. Task is a profile, not yet repeatable. |
| L6 | AUTO-READY | All RTP fields filled. Automation requirements documented. Ready to build. |
| L7 | AUTO-BUILT | Automation exists (script/cron/agent). Needs testing. |
| L8 | AUTO-TESTING | Running with human verification. Tracking success rate. |
| L9 | AUTONOMOUS | 90%+ success over 5+ runs. Human only for exceptions. |

**Source:** CMMI maturity model, automation pyramid, progressive automation.
**System rule:** Maturity can drop (if automation breaks → L8/L7). Assessed during daily scan (CLAWD-004).

## Tracing

### Top-Down Trace
**Definition:** Starting from the goal, drilling down through sub-goals → mechanism chain → bottleneck → paths → tasks to find what's blocking progress. Answers: "Given my goal, what should I work on?"
**Example:** GOAL-001 → SG-001 → Mechanism chain → Bottleneck at M2 → Shared requirements incomplete → BC-OFFER-V1 is the root blocker.
**Source:** Hoshin Kanri (catchball), root cause analysis, Theory of Constraints (find the constraint).
**System rule:** Top-down trace follows hard dependencies only for "what's physically blocking." Reports soft dependencies separately.

### Bottom-Up Trace
**Definition:** Starting from any task, tracing upward through path → mechanism step → sub-goal → goal. Answers: "Why am I doing this task? What goal does it serve?"
**Example:** LP-003 → PATH-LI → M1: Reach → SG-001 → GOAL-001. "I'm writing intros because it feeds reach, which feeds client acquisition, which feeds the £5k goal."
**Source:** Traceability (systems engineering), purpose alignment.
**System rule:** Every task must trace to a goal in ≤4 hops. If it can't → orphaned task → either link it or question its existence.

### Sense-Check
**Definition:** Periodic verification that execution is aligned with strategy. Asks: "Are the tasks we're doing actually serving the current bottleneck?" Catches drift where busy-work replaces bottleneck-focused work.
**Example:** "We're building Reddit outreach (PATH-RD) but the bottleneck is at M2 (Engage), not M1 (Reach). Adding more reach paths is waste — fix the booking link first."
**Source:** Hoshin Kanri review, strategy-execution alignment audit, OODA loop (Orient phase).
**System rule:** Part of the strategy review cadence. Uses top-down trace to verify current work targets the bottleneck.

### Cadence
**Definition:** A time-based rhythm for reviewing and updating different layers of the system. Cadences prevent both neglect (never reviewing) and flip-flopping (reviewing too often). Different layers have different cadences.
**Example:** Daily: execution review (what did I do, what's next). Weekly: path/mechanism review (is the bottleneck moving?). Monthly: goal/sub-goal review (is the strategy right?).
**Source:** Hoshin Kanri (annual → quarterly → monthly → daily), Scrum ceremonies, PDCA cycle timing.
**System rule:** See System Modes below for when each cadence fires.

### Trigger
**Definition:** An event-based state change, as opposed to a time-based cadence. When a specific condition occurs, something changes state automatically — no waiting for the next review cycle.
**Example:** "When a threshold gate's metric reaches 100% → unlock downstream work immediately, don't wait for weekly review."
**Source:** Event-driven architecture, state machines, interrupt-based processing.
**System rule:** Triggers are defined on threshold gates and scanner alerts. The scanner checks conditions; when met, it fires the trigger.

## System Modes

### Execution Mode
**Definition:** "Just do the work." Morning mode. No strategic questioning, no system redesign. Pick the next task from the current bottleneck-aligned work and execute. Trust the plan.
**When:** Mornings, weekday working hours.
**Behaviour:** Top-down trace → find highest-priority unblocked task → do it. Don't question whether the goal is right.
**Source:** Deep work (Newport), time-boxing, OODA loop (Act phase).

### Strategy Mode
**Definition:** "Think about the plan." Review whether the strategy is working, whether the bottleneck has moved, whether paths should be added/dropped. This is when goals, mechanism chains, and paths get updated.
**When:** Evenings, weekends, designated review sessions.
**Behaviour:** Sense-check → update mechanism chain ratios → reassess bottleneck → adjust paths → update goal file.
**Source:** Hoshin Kanri review, OODA loop (Observe/Orient), retrospective (Scrum).
**System rule:** Strategy changes ONLY happen in strategy mode. If an insight occurs during execution mode → capture it as a note, don't act on it until strategy mode.

### Build Mode
**Definition:** "Creating the system." Building new infrastructure, processes, tools. Active construction with high cognitive load and frequent decisions. Has a defined end point (system stop criteria).
**When:** System Status = BUILDING. Typically early stages of a new goal or major system change.
**Behaviour:** Create templates, build automation, define processes, set up tools. Heavy investment now for reduced investment later.
**Source:** Project management (PMBOK), infrastructure engineering.
**System rule:** Build mode should always have stop criteria. If you're building without knowing when to stop, you're over-engineering.

### Run Mode
**Definition:** "Operating the system." System is built and self-sustaining. Operator monitors dashboards, handles exceptions, makes minor adjustments. Low cognitive load, predictable time investment.
**When:** System Status = OPERATIONAL. System stop criteria met.
**Behaviour:** Monitor → detect anomalies → intervene only when needed. Routine tasks are automated (L7+). Human handles exceptions and strategic reviews only.
**Source:** ITIL (service operation), DevOps (run books), operations management.
**System rule:** Run mode time investment should be <2hrs/day for any single goal's system. If it's more → something isn't automated that should be, or the system isn't truly operational.

## Cross-Cutting Concepts

### Shared Requirement
**Definition:** A goal-level condition that must be true for any path to convert through a mechanism step. Not a task itself — it's a *condition* implemented by one or more tasks. Shared requirements block at the mechanism level, not the task level.
**Example:** "Coaching offer defined" — implemented by task BC-OFFER-V1. Blocks M2 and M3. No path can convert without it.
**Source:** Infrastructure prerequisites, platform capabilities, cross-cutting concerns (software architecture).
**System rule:** Separate entity type from tasks. Status derived from implementing tasks' statuses. Surfaced in goal file's Shared Requirements table.

### Minimum Viable Path (MVP)
**Definition:** The single path with the fewest requirements and fastest route to revenue. Focus here first. Other paths are deferred until the MVP path proves the mechanism chain works.
**Example:** PATH-LI (LinkedIn Outreach) — already partially built, furthest along, fewest new requirements.
**Source:** Lean startup (MVP), focused strategy, Theory of Constraints (don't spread effort across non-constraints).
**System rule:** One path marked `isMVP: true` per sub-goal. All shared requirements must be met for the MVP path before adding other paths.

### Profile vs Instance
**Definition:** A **profile** is a task type that persists and evolves (e.g., "Write LinkedIn Intros"). An **instance** is a single execution of that profile (e.g., "Write intros on 2026-02-20"). Profiles have versioned processes. Instances are rows in the execution log, linked to the version they ran under.
**Source:** Object-oriented design (class vs instance), process management.
**System rule:** Instances never overwrite profiles. They're evidence. If an instance reveals the process needs changing → version bump on the profile.
