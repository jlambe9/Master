## 1. Strategy/Execution Separation

### Framework: Hoshin Kanri (方針管理)
**Source:** Toyota, 1960s. Yoji Akao. Widely used in Lean management.
**Key Insight:** Strategy is deployed top-down through "catchball" — each level translates the level above into actionable targets, then pushes back if the target is unrealistic. This creates alignment WITHOUT micromanagement. Critically: **strategy is set annually, reviewed quarterly, and execution is reviewed monthly/daily.** The different cadences prevent flip-flopping.

**Cadence structure:**
- **Annual:** Set 3-5 breakthrough objectives (not 20)
- **Quarterly:** Review progress against annual objectives. Adjust tactics, NOT objectives (unless truly broken)
- **Monthly:** Review leading indicators. Are mechanism steps converting?
- **Daily:** Execute. Don't question the strategy. Just do the work.

**Anti-flip-flop mechanism:** The rule is: **you cannot change the objective at a lower cadence than the one that set it.** Daily execution can't override monthly tactics. Monthly reviews can't override quarterly strategy. This is the separation.

**Application to Jamie's system:** Maps directly. Goals = annual/quarterly hoshin. Mechanism chains = the deployment. Paths = tactics. Tasks = daily execution. The cadence structure should be:
- Goals reviewed monthly at most (strategy mode)
- Mechanism chains / bottleneck reviewed weekly (strategy mode)
- Paths reviewed weekly (strategy mode)
- Tasks executed daily (execution mode)

**Adopt terminology?** Partially. "Hoshin" itself is jargon. But adopt the **cadence discipline** — it's the most valuable part. Also adopt the principle: "you can't change strategy from execution mode."

---

### Framework: Balanced Scorecard (BSC)
**Source:** Kaplan & Norton, 1992. Harvard Business Review.
**Key Insight:** Measures strategy across four perspectives: Financial, Customer, Internal Process, Learning & Growth. The breakthrough: **leading indicators (process, learning) predict lagging indicators (financial, customer).** You manage by leading indicators and verify with lagging ones.

**How it prevents flip-flopping:** By defining cause-and-effect chains. "If we improve [process metric], it will improve [customer metric], which will improve [financial metric]." Changes to strategy must show which causal link is broken, not just "revenue is low."

**Application to Jamie's system:** The mechanism chain IS a balanced scorecard cause-and-effect chain. M1 throughput (leading) → M2 conversion (leading) → M3 close rate (leading) → Revenue (lagging). Jamie should manage daily by M1/M2 throughput, not by revenue.

**Adopt terminology?** Yes: **leading indicator** and **lagging indicator** are universally understood and should be in the lexicon. Don't adopt the four-perspective structure — too corporate for a solo business.

---

### Framework: OODA Loop
**Source:** Colonel John Boyd, USAF, 1960s-80s. Military strategy.
**Key Insight:** Decision-making as a loop: Observe → Orient → Decide → Act. The side that cycles through OODA faster wins. But crucially: **Orient is where strategy lives.** Observation without good mental models produces bad decisions. And: **you can loop at different speeds for different decisions.**

**How it prevents flip-flopping:** It doesn't, by itself. OODA is fast adaptation. But Boyd's insight is that **Orient (your mental model) should change slowly while Act changes fast.** Tactics are fast OODA. Strategy is slow OODA. The mistake is running strategy at tactical speed.

**Application to Jamie's system:** Execution mode = fast OODA (observe task result → orient against path goals → decide next task → act). Strategy mode = slow OODA (observe mechanism chain metrics → orient against goal → decide path changes → act on system). Different loop speeds for different layers.

**Adopt terminology?** No as formal terms (too military/abstract). But adopt the **principle**: fast loops for tactics, slow loops for strategy. This maps to our execution mode vs strategy mode.

---

### Synthesis for Jamie's System

**Recommendation:** Adopt hoshin kanri's cadence discipline + BSC's leading/lagging distinction + OODA's multi-speed loops. Concrete implementation:

| Layer | Review Cadence | Mode | Can Change |
|-------|---------------|------|------------|
| Goal | Monthly | Strategy | Objective, metric, deadline |
| Sub-Goal | Monthly | Strategy | Targets, add/remove sub-goals |
| Mechanism Chain | Weekly | Strategy | Ratios, bottleneck assessment |
| Path | Weekly | Strategy | Add/pause/drop paths |
| Task | Daily | Execution | Priority, assignments, next action |

**Rule:** A daily insight about strategy gets CAPTURED as a note. It gets ACTED ON in the next strategy review. Not before.

## 2. Mechanism Chain Structure — Sales/Marketing Funnel Terminology

### Framework: AIDA (Attention, Interest, Desire, Action)
**Source:** E. St. Elmo Lewis, 1898. Advertising theory.
**Key Insight:** The oldest funnel model. Buyers move through psychological stages. Each stage has a different conversion mechanism.
**Terminology:** Attention → Interest → Desire → Action.
**Limitation:** Linear, assumes passive buyer, no post-sale stages.

### Framework: TOFU/MOFU/BOFU
**Source:** HubSpot / inbound marketing, 2000s.
**Key Insight:** Funnel as three zones: Top of Funnel (awareness), Middle of Funnel (consideration), Bottom of Funnel (decision). Content and tactics differ by zone.
**Terminology:** TOFU → MOFU → BOFU.
**Limitation:** Content-marketing centric. Assumes inbound. Doesn't fit outbound well.

### Framework: Revenue Operations (RevOps) Pipeline
**Source:** Salesforce, HubSpot, modern B2B SaaS, 2010s.
**Key Insight:** The modern standard for B2B pipeline stages. Not about psychological states but about **observable actions** (which is what Jamie needs — measurable conversions):

| Stage | Observable Action | Conversion Question |
|-------|-------------------|---------------------|
| **Lead** | Contact identified, info captured | Do we know who they are? |
| **MQL** (Marketing Qualified Lead) | Engaged with content, meets criteria | Have they shown interest? |
| **SQL** (Sales Qualified Lead) | Discovery call booked/completed | Are they a fit? |
| **Opportunity** | Proposal/offer presented | Do they want what we sell? |
| **Closed Won** | Contract signed, payment received | Did they buy? |
| **Closed Lost** | Declined, went elsewhere, ghosted | Why not? |

**Limitation:** Assumes marketing/sales team division. Solo operator doesn't have MQL/SQL distinction.

### Framework: Pirate Metrics (AARRR)
**Source:** Dave McClure, 500 Startups, 2007.
**Key Insight:** Startup growth metrics: Acquisition → Activation → Retention → Revenue → Referral. Covers the FULL lifecycle including post-sale.
**Terminology:** AARRR.
**Limitation:** Product/SaaS focused. "Activation" doesn't map well to services.

### Is "Distribution → Conversion → Close" the Standard?

**No.** Jamie's current "Reach → Engage → Close" is a simplified three-stage model. The standard terminology varies by framework, but the most universal B2B stages are:

1. **Prospecting / Outreach** — finding and contacting potential clients
2. **Qualification / Discovery** — determining fit through conversation
3. **Proposal / Presentation** — presenting the offer
4. **Negotiation / Objection Handling** — addressing concerns
5. **Close** — contract signed, payment received

For Jamie's solo coaching business, a three-stage model is appropriate (he doesn't have separate marketing/sales teams), but the naming should be more precise:

### Recommendation for Jamie's System

| Jamie's Current | Proposed | Why |
|----------------|----------|-----|
| M1: Reach | M1: **Outreach** | "Reach" is ambiguous (impressions? contacts?). "Outreach" = active contact initiation. |
| M2: Engage | M2: **Qualify** | "Engage" is vague. The actual conversion is: contact → qualified prospect (someone who fits AND is interested). This includes conversation + discovery call. |
| M3: Close | M3: **Close** | Keep. Universally understood. Includes: present offer, handle objections, sign contract, receive payment. |

**Alternative 4-stage model** (if M2 proves too broad):
M1: Outreach → M2: Conversation → M3: Discovery Call → M4: Close

**Adopt when:** Data shows a meaningful drop-off within M2 that needs separate tracking (e.g., plenty of conversations but zero calls booked → split M2 into Conversation + Call Booked).

**Terminology adoption:** Use **Outreach → Qualify → Close** as mechanism step names. Use **pipeline** (not "funnel") — pipeline implies active management of individual leads; funnel implies passive flow of anonymous traffic. Jamie is doing pipeline work.

## 3. "Stick to Plan vs Change Plan" Decision Framework

### Framework: PDCA (Plan-Do-Check-Act)
**Source:** W. Edwards Deming, 1950s (building on Shewhart cycle).
**Key Insight:** Continuous improvement as a cycle. The critical insight for stick-vs-change: **Check comes before Act.** You don't just DO and then CHANGE. You DO, then CHECK (did it work?), THEN decide whether to ACT (standardise what works or adjust what doesn't).

**Decision rule:**
- **Check says "working"** → Standardise. Don't change.
- **Check says "not working"** → Analyse WHY before changing. Was execution poor (→ execute better) or was the plan wrong (→ change plan)?

### Framework: OODA Loop — Boyd's Law
**Source:** John Boyd.
**Key Insight:** "He who can handle the QUICKEST rate of change survives." But this is about TACTICAL speed, not STRATEGIC speed. Boyd's actual teaching: **orient correctly first (slow), then act decisively (fast).** The mistake people make: they re-orient every cycle, which looks like flip-flopping.

**Boyd's real decision framework:**
1. Is my mental model of the situation still valid? (Orient)
2. If yes → act faster, don't change direction.
3. If no → update mental model, THEN change direction.
4. Signal that your model is wrong: **surprising observations.** Expected inputs producing unexpected outputs.

### Framework: Amazon's "Disagree and Commit"
**Source:** Jeff Bezos, Amazon leadership principles.
**Key Insight:** Once a decision is made, EVERYONE commits to execution even if they disagreed. Review happens at predetermined checkpoints, not whenever someone feels uncomfortable. This prevents the "but I still think we should..." erosion of execution.

**For solo operators:** You disagree with YOURSELF. Your morning execution self disagrees with your evening strategic self. The discipline: **commit to the plan during execution mode. Capture doubts as notes. Review doubts in strategy mode only.**

### Framework: Real Options Theory
**Source:** Financial engineering applied to project management (Stewart Myers, 1977; adapted by Chris Matts).
**Key Insight:** Decisions are like financial options — they have value, and you should delay irreversible decisions until the "last responsible moment." But once you commit, you commit fully.

**Decision types:**
- **Reversible (Type 2, per Bezos):** Make fast, change fast. Most path-level decisions. Adding a new outreach channel is reversible.
- **Irreversible (Type 1):** Deliberate, get data first. Goal-level changes. Pivoting from coaching to consulting is hard to reverse.

### Synthesis: When to Adapt vs Persevere

**Signals that you should PERSEVERE (execute harder):**

| Signal | Meaning |
|--------|---------|
| Leading indicators are moving in the right direction | The strategy is working, just slowly |
| You haven't given it enough time (less than 2 full PDCA cycles) | Insufficient data to judge |
| The bottleneck is at execution, not strategy | You know WHAT to do, you're just not doing enough of it |
| Your doubts are emotional, not data-driven | FEAR/PANIC system, not genuine strategic insight |
| You keep changing plans before completing one cycle | This IS the problem — you're not executing, you're deliberating |

**Signals that you should CHANGE THE PLAN:**

| Signal | Meaning |
|--------|---------|
| Leading indicators flat or declining after 2+ cycles | The mechanism isn't converting despite execution |
| Surprising observations (expected input → unexpected output) | Your model of reality is wrong |
| External environment changed (market, competition, regulation) | The terrain shifted, old map is wrong |
| Consistent feedback from prospects/clients contradicts assumptions | Direct evidence your value proposition is off |
| The bottleneck is structural, not effort-based | More effort won't fix a broken mechanism step |
| Cost of continuing > cost of pivoting | Sunk cost fallacy check |

**Application to Jamie's system:**

Implement a **strategy change protocol:**

1. **Capture the doubt** — write it in the goal file's decision log with today's date
2. **Wait for strategy mode** — don't act on it during execution mode
3. **Check the data** — in strategy mode, look at leading indicators for the relevant mechanism step
4. **Apply the signals table** — does the data say persevere or change?
5. **If change:** document the decision in the decision log with rationale. Version-bump the affected level (path change = minor, mechanism change = major, goal change = new goal).
6. **If persevere:** document WHY you're persevering. Set a review date. Execute without second-guessing until that date.

**Adopt terminology?** Yes: adopt **PDCA** as the explicit improvement cycle. Adopt **Type 1/Type 2 decisions** (irreversible/reversible) as a shorthand. Adopt the **signals table** as a decision aid.

## 4. Business Primitives / Infrastructure Components

### Framework: Business Model Canvas (BMC)
**Source:** Alexander Osterwalder & Yves Pigneur, *Business Model Generation*, 2010.
**Key Insight:** Nine building blocks that describe how a business creates, delivers, and captures value. These are the closest thing to "irreducible components":

| Block | Definition | Jamie's Equivalent |
|-------|-----------|-------------------|
| **Customer Segments** | Who you serve | ADHD professionals/entrepreneurs |
| **Value Propositions** | What value you deliver | ADHD coaching — structured support for executive function |
| **Channels** | How you reach and deliver | LinkedIn (PATH-LI), Meetups (PATH-MU), etc. |
| **Customer Relationships** | Type of relationship | 1:1 coaching sessions |
| **Revenue Streams** | How you earn money | Monthly coaching retainer (£500+/month) |
| **Key Resources** | What you need to operate | Jamie's expertise, scheduling system, coaching methodology |
| **Key Activities** | What you must do | Outreach, qualify, coach, retain |
| **Key Partnerships** | Who helps you | None currently (solo) |
| **Cost Structure** | What it costs | Tools (Expandi, PB, APIs), time |

**Application:** The BMC blocks map to Jamie's system as:
- Customer Segments + Value Propositions → **Goal definition** (who and what)
- Channels → **Paths**
- Key Activities → **Mechanism chain steps**
- Key Resources → **Shared Requirements / Business Primitives**
- Revenue Streams → **Goal metric**
- Cost Structure → **Sub-goal SG-002 (expenses)**

### Framework: Lean Canvas
**Source:** Ash Maurya, *Running Lean*, 2012. Adapted from BMC for startups.
**Key Insight:** Replaces BMC's "Key Partners" and "Customer Relationships" with **Problem**, **Solution**, and **Unfair Advantage** — more relevant for early-stage businesses. Also adds **Key Metrics**.

| Lean Canvas Block | Jamie's Equivalent |
|-------------------|-------------------|
| Problem | ADHD professionals struggling without support |
| Solution | Structured coaching with accountability |
| Key Metrics | Leading indicators in mechanism chain |
| Unfair Advantage | Jamie's lived experience + professional training |
| Channels | Paths |
| Cost Structure | SG-002 |
| Revenue Streams | Goal metric |

### Framework: Value Proposition Canvas
**Source:** Osterwalder, *Value Proposition Design*, 2014.
**Key Insight:** Zooms into the Customer Segment ↔ Value Proposition fit. Two sides: **Customer Profile** (jobs, pains, gains) and **Value Map** (products/services, pain relievers, gain creators). The fit between them IS product-market fit.

**Application:** This is what BC-OFFER-V1 should define. The offer document should explicitly map:
- Client jobs → Jamie's coaching activities that address them
- Client pains → Jamie's pain relievers
- Client gains → Jamie's gain creators

### What Are the Irreducible Components (Business Primitives)?

Synthesising across frameworks, the irreducible components for ANY business are:

| Primitive | Definition | Jamie's Instance | System Mapping |
|-----------|-----------|-----------------|----------------|
| **Offer** | What you sell, to whom, for how much | ADHD coaching, £500+/month | Shared Requirement → BC-OFFER-V1 |
| **Audience** | Who you serve + how to find them | ADHD professionals on LinkedIn | Path targeting criteria |
| **Channel** | How you reach the audience | LinkedIn, Meetups, Reddit | Paths |
| **Conversion Mechanism** | How interest becomes revenue | Outreach → Qualify → Close | Mechanism chain |
| **Delivery System** | How you deliver the value after sale | 1:1 coaching sessions (Zoom?) | Post-close tasks (BOSS-ONBOARD) |
| **Revenue Model** | How money flows in | Monthly retainer | Payment system (Stripe) |
| **Legal Framework** | Permission to operate | Contract, T&Cs, GDPR | Shared Requirements |
| **Cost Structure** | What it costs to operate | Tools, time | FIN-001 |

**Adopt terminology?** Yes: adopt **Business Primitive** as the term for these irreducible components. Each primitive maps to either a Shared Requirement (in the goal file) or a fundamental system assumption. If any primitive is missing, the business literally cannot function.

**Recommendation:** Add a "Business Primitives" section to the goal file template that lists the 8 primitives above with their current status. This replaces the ad-hoc shared requirements with a structured checklist.

## 5. Run Rate / Operational Readiness

### Framework: ITIL Service Operation
**Source:** ITIL (Information Technology Infrastructure Library), UK Government/Axelos.
**Key Insight:** Distinguishes between **service transition** (build/deploy) and **service operation** (run/maintain). The transition to operations requires formal handover with documented runbooks, monitoring, and escalation paths. Key concept: **a service is operational when it can be run by someone other than the person who built it.**

**Metrics for operational status:**
- Documented runbook exists
- Monitoring/alerting active
- Known error database (what can go wrong + how to fix)
- Incident response process defined
- Mean Time Between Failures (MTBF) acceptable
- Mean Time To Recovery (MTTR) acceptable

**Application:** Jamie's system should be "operational" when Rich (or a future assistant) can run the daily pipeline without Jamie except for human-required steps. The System Stop Criteria checklist is essentially an ITIL operational readiness review.

### Framework: DevOps Maturity / SRE
**Source:** Google SRE (Site Reliability Engineering), 2016.
**Key Insight:** Defines **Service Level Objectives (SLOs)** — the target reliability/performance for a service. When the service meets its SLOs consistently, it's in run mode. When it doesn't, it needs engineering (build mode). The **error budget** concept: you're allowed some failures. If you're within budget → run mode. If you're burning through budget → build mode (fix things).

**Application to Jamie's system:**
- Define SLOs for each automated process: "Intro writing succeeds 90% of the time without intervention"
- Track error rate. Below threshold → run mode. Above → investigate and fix.
- This maps directly to the L8 → L9 maturity transition (90%+ success rate).

### Framework: Toyota Production System — Standard Work
**Source:** Taiichi Ohno, Toyota, 1950s-80s.
**Key Insight:** A process is in "standard work" when: (1) the sequence is defined, (2) the time per cycle is known, (3) the work-in-progress is controlled, (4) anyone trained can perform it. Standard work is the DEFINITION of run mode. Deviations from standard work trigger improvement cycles (kaizen), not panic.

**Metrics for maintenance mode:**
- Cycle time is stable and predictable
- Output quality is consistent
- WIP is controlled (not growing unboundedly)
- Exceptions are rare and have documented responses
- Continuous improvement is incremental, not structural

### Framework: Operational Readiness Review (ORR)
**Source:** NASA, military, aerospace.
**Key Insight:** Before a system "goes live," a formal review verifies ALL prerequisites are met. The ORR checklist covers: hardware ready, software tested, personnel trained, procedures documented, monitoring active, contingency plans in place. If ANY item fails → system is NOT operationally ready.

**Application:** Jamie's domain operational checklists (in task-field-guide.md) ARE mini-ORRs. They should be treated as formal gate checks: if any item fails, the domain is NOT in run mode, regardless of how many tasks are "completed."

### Synthesis: What Defines "Self-Sustaining"?

A system is in maintenance/run mode when ALL of the following are true:

| Criterion | Metric | Jamie's System Check |
|-----------|--------|---------------------|
| **Documented** | Runbook/process exists for every step | All RTP fields filled (L6+) |
| **Automated** | Automatable steps are automated | Maturity L7+ for automatable tasks |
| **Reliable** | 90%+ success rate for automated steps | L9 maturity achieved |
| **Monitored** | Alerts fire when something breaks | Scanner/cron detecting failures |
| **Time-bounded** | Steady-state time < threshold | <2hrs/day total across all paths |
| **Exception-based** | Human intervenes only for exceptions | Most daily work is monitoring, not doing |
| **Predictable** | Output rate is stable week-over-week | Mechanism step throughput has low variance |

**When to declare "operational":**
- All System Stop Criteria met ✅
- All domain checklists passing ✅
- All automatable tasks at L7+ ✅
- Total daily time investment < 2hrs ✅
- At least 2 weeks of stable operation ✅

**Adopt terminology?** Yes:
- **Operational Readiness** — the gate check before declaring run mode (adopt from NASA/ORR)
- **Standard Work** — the documented, repeatable process that defines run mode (adopt from Toyota)
- **SLO** — the success rate target for automated processes (adopt from SRE, but don't use the acronym in daily use — say "success target")
- **Run Mode / Build Mode** — the two system states (already in the lexicon, validated by all frameworks)
