# v1.3 Critique — Strategy vs Execution Separation

> **Reviewer:** ATLAS-4 (Business Systems Architect)
> **Angle:** Strategic vs Execution layer boundary
> **Date:** 2026-02-22
> **Status:** COMPLETE
> **Reviewed:** v1.3-goal-framework-draft.md

---

## Verdict: The Boundary is Clean in Theory, Leaky in Practice

The draft correctly identifies the two layers and places the **Path** entity as the bridge. The hierarchy (`Goal → Sub-Goal → Mechanism Step || Path → Task → Subtask`) is sound. But several design choices create bleed points where strategy leaks into execution or vice versa, and several operational questions are unanswered.

---

## 1. Boundary Analysis: Where It's Clean, Where It Leaks

### ✅ Clean

- **Mechanism Steps are quantities, not work.** "Speak to 100 people" is a throughput target, not a task. This is the right abstraction.
- **Tasks don't change.** v1.2 task files stay task files. No strategic metadata injected into them.
- **Paths are interchangeable.** Swapping LinkedIn for Reddit doesn't touch the mechanism chain. Good.

### ⚠️ Leaky

**Leak 1: Shared Requirements are strategy-level entities implemented by execution-level tasks.** The `IMPLEMENTED_BY` edge crosses the boundary. This is *necessary* — requirements must be satisfied by work — but it means the goal file contains task IDs (`BC-OFFER-V1`, `BOSS-GDPR`). When those tasks change ID, get split, or get replaced, the goal file needs updating. This is coupling.

**Mitigation:** Accept it. The alternative (indirection through another entity) is worse. But document the maintenance rule: when a task implementing a shared requirement changes, update the goal file. The sync script should flag orphaned `IMPLEMENTED_BY` edges.

**Leak 2: The Path table in the goal file lists task ID ranges (`LP-001 → LP-006`).** This is execution detail in a strategy document. When tasks are added/removed from a path, the goal file needs editing. For 1-2 goals this is fine. For 5+ goals it becomes a maintenance burden.

**Mitigation:** Remove task ID ranges from the goal file. The Path just needs an ID and status. Tasks-within-path is a query: `MATCH (p:Path {id:'PATH-LI'})-[:CONTAINS_TASK]->(t) RETURN t`. Or a link to the path's section in the task registry. Don't duplicate it.

**Leak 3: "Bottleneck Action" in the goal file links to tasks/paths.** Same issue as Leak 2 — execution references in strategy. This one is harder to remove because the whole point is "what are we doing about the bottleneck?" 

**Mitigation:** Keep it, but as a free-text note pointing to a path, not specific task IDs. "Currently addressed by PATH-LI tasks" not "LP-003, LP-004."

## 2. Morning Plan: What Does Jamie See?

**The draft doesn't specify this, and it should.**

When the daily planning skill runs at 21:00 and produces the 3-bucket plan for tomorrow, what layer does it query?

**What Jamie should see at planning time (my recommendation):**

```
## Tomorrow's Focus

### Strategic Context (from Goal layer)
- GOAL-001: £5k/month — BUILDING
- Bottleneck: M2 (Engage) — 0 calls booked
- Blocker: Shared requirements 0/5 complete
- MVP path: LinkedIn (ACTIVE, 2/4 requirements met)

### Execution Plan (from Task layer)
🔴 Must Do:
- BC-OFFER-V1: Define coaching offer (blocks everything downstream)

🟡 Should Do:
- LP-005: Build conversation-to-call process

🟢 Could Do:
- LP-006: Optimise CR automation
```

**The rule:** Strategic context is a *header* — 3-5 lines max, read-only, orienting. Execution plan is where decisions happen. Jamie never *edits* strategy during morning planning. He edits it during weekly/monthly reviews.

**What needs specifying:**
- The daily planning skill needs to READ goal files for context but OUTPUT only task-level actions.
- Goal-level metrics should surface as a dashboard line, not as actionable items (unless a threshold gate fires — see §4).
- The plan should trace each task back to its mechanism step in one parenthetical: `BC-OFFER-V1: Define coaching offer (blocks M2/M3)`.

## 3. "Build in Mud" — Walkthrough

**Scenario:** Jamie has GOAL-001. PATH-LI (LinkedIn) is the MVP. After 4 weeks, LinkedIn outreach yields 200 contacts, 2 conversations, 0 calls. The 5% conversion ratio is actually 1%. Jamie decides LinkedIn cold outreach isn't working and pivots to PATH-MU (Meetups).

### What changes:

| Entity | Change | File Impact |
|--------|--------|-------------|
| PATH-LI | Status: ACTIVE → PAUSED (or DROPPED) | Goal file: Paths table row updated |
| PATH-LI tasks (LP-001→006) | Status: various → PAUSED | Individual task files updated |
| PATH-MU | Status: NOT STARTED → ACTIVE, isMVP: true | Goal file: Paths table row updated |
| PATH-LI | isMVP: false | Goal file: Paths table row updated |
| M1: Reach | conversionRatio updated from 5% → 1% (LinkedIn data) | Goal file: Mechanism Chain table |
| New tasks | MU-001, MU-002, etc. created for meetup path | New task files |
| Decision Log | "Pivoted from LinkedIn to Meetups because..." | Goal file: Decision Log table |

### What stays:
- **Goal** — unchanged (still £5k/month)
- **Sub-Goals** — unchanged
- **Mechanism Steps** — same chain (Reach → Engage → Close). Ratios may update.
- **Shared Requirements** — unchanged (offer, booking, payment still needed regardless of path)
- **Threshold Gates** — unchanged

### What this validates:
The framework handles pivots well. Strategy layer is untouched. Only Path status and MVP designation change. The mechanism chain absorbs new conversion data without restructuring. The "build in mud" principle works because **paths are disposable but mechanism steps are durable.**

### What this reveals as a gap:
**There's no "path retrospective" mechanism.** When PATH-LI gets dropped, the *learnings* (actual conversion ratios, what didn't work, time invested) should be captured somewhere persistent. The Decision Log captures the pivot decision, but not the quantitative data. 

**Recommendation:** Add a `## Path Archive` section to the goal file. When a path is DROPPED/PAUSED, move its row to the archive with final metrics and a one-line retrospective. This is strategy-layer data (conversion evidence), not execution.

## 4. Threshold Gates: Firing Mechanism

**The draft defines threshold gates but doesn't specify HOW they fire.** This is the biggest operational gap.

### Questions the draft doesn't answer:

1. **Who checks?** A cron job? The daily planning skill? Jamie manually?
2. **How often?** Every planning cycle? On every task completion? Real-time?
3. **What happens when one fires?** Does it auto-create tasks? Send a notification? Just change status to UNLOCKED?
4. **Where does `currentValue` come from?** Manual update? H1 sheet? Neo4J query?

### My recommendation:

**Checking cadence:** Weekly, during a "goal review" cron or skill. NOT daily — threshold gates are strategic, not execution-level. Checking daily creates noise.

**Firing action:**
1. Gate status changes: LOCKED → UNLOCKED
2. Notification sent to Jamie: "TG-001 unlocked: ≥3 clients signed. You can now: Build group program structure."
3. **No auto-task-creation.** The gate UNLOCKS the *option* to do something. Jamie decides whether and when to act on it. This preserves the "don't think beyond current scope" principle.

**Metric source:** Each gate needs a `source` field specifying exactly where to read the current value. If it's a H1 column, specify which. If it's a Neo4J query, specify which. Vague sources = gates that never fire because nobody checks.

**Add to the template:**
```markdown
| Gate | Condition | Current | Source | Check Frequency | Unlocks | Status |
```

## 5. Stop Criteria: Enforceable or Aspirational?

**Currently aspirational.** The draft lists checkboxes but provides no enforcement mechanism.

### The problem:

Take: `"Automatable tasks automated (≥L7)"`. Who checks this? How? The system doesn't currently track automation levels across all tasks. This criterion is unmeasurable without tooling that doesn't exist yet.

Take: `"Steady-state time commitment: <2hrs/day"`. How is this measured? Time tracking? Self-report? This is aspirational unless there's a mechanism to actually measure time spent.

### What's enforceable today:

| Criterion | Enforceable? | How |
|-----------|-------------|-----|
| All paths mapped with tasks | ✅ Yes | Query: any path with 0 tasks? |
| MVP path fully operational | ✅ Yes | Query: all tasks in MVP path ≥ IN PROGRESS? |
| Shared requirements complete | ✅ Yes | Query: all SRs status = COMPLETE? |
| Automatable tasks ≥L7 | ❌ No | No automation level tracking exists |
| <2hrs/day steady state | ❌ No | No time tracking mechanism |
| Monitoring/alerts running | ⚠️ Partial | Can check if cron jobs exist, but "monitoring" is vague |
| Bottleneck has active intervention | ✅ Yes | Query: bottleneck step has ≥1 active path with active tasks? |

### Recommendation:

1. **Drop unmeasurable criteria or defer them.** If you can't query it, it's not a stop criterion — it's a wish.
2. **Make each criterion a query.** If it can't be expressed as a Neo4J or file-system query, it's not machine-enforceable.
3. **System Status transitions should be gated by enforceable criteria only:**
   - BUILDING → OPERATIONAL: All SRs complete + MVP path has all tasks ≥ IN PROGRESS + bottleneck has intervention
   - OPERATIONAL → OPTIMISING: ≥1 client signed + all enforceable stop criteria met
4. **The aspirational ones (time commitment, full automation) become goals of their own** rather than system stop criteria.

## 6. Daily Planning Skill — Data Flow

The daily planning skill (21:00, 3-bucket output) needs to be explicitly designed for two-layer reading:

### Input Data:

**From Goal Layer (read-only context):**
- Active goals and their system status (BUILDING/OPERATIONAL/OPTIMISING)
- Current bottleneck per goal
- Shared requirement status (what's blocking)
- Threshold gates near firing (≥80% of threshold)
- MVP path designation

**From Task Layer (actionable items):**
- Tasks in ACTIVE paths with status CAPTURED or IN PROGRESS
- Task dependencies (what's unblocked today)
- Task priority scores (existing v1.2 Layer 3)
- Subtask completion status

### Output Format (recommended):

```markdown
# Plan — 2026-02-23

## Context (auto-generated, read-only)
GOAL-001 [BUILDING]: Bottleneck at M2 (Engage). 0/5 shared requirements done.
→ Nothing converts until offer + booking exist. Pipeline work is premature.

## 🔴 Must Do (1-2 items)
- [ ] BC-OFFER-V1: Define coaching offer → blocks M2, M3
  - Why: Nothing downstream works without this

## 🟡 Should Do (2-3 items)
- [ ] BOSS-GDPR: Legal framework → blocks M3
  - Why: Needed before first client can onboard

## 🟢 Could Do (1-2 items)
- [ ] LP-005: Conversation-to-call process → feeds M2
  - Why: Useful to draft now, but can't test until booking link exists

## ⏸️ Explicitly NOT Doing
- PATH-MU, PATH-RD, PATH-REF: Don't create tasks for these. MVP isn't operational yet.
- TG-001 items: 0/3 clients. Group program is locked. Don't think about it.
```

### Key design principle:
The **"Explicitly NOT Doing"** section is how the system enforces "don't think beyond current scope." It's not just absence of tasks — it's active deferral, made visible so Jamie doesn't wonder "should I be working on meetups?"

## 7. Enforcing "Don't Think Beyond This"

Jamie said: *"We don't really need to think beyond this until we get to this state."*

### The draft's implicit mechanism: Threshold Gates.
If something is behind a LOCKED gate, it shouldn't be planned for. This is correct but passive — it relies on the planner knowing to check gates.

### What's missing: Active deferral.

**Current state:** "Don't create the task file yet" = invisible deferral. There's no record that something was *considered and deferred*, only that it doesn't exist. This means:
- Jamie might re-think it from scratch later (wasted cognition)
- There's no place to dump half-formed ideas about future work
- The system can't surface "this was deferred, now it's unlocked" because there's no deferred entity to unlock

### Recommendation: "Deferred Work" as a section in the Goal file.

```markdown
## Deferred Work (Unlocked by Gates)

| Idea | Gate | Notes |
|------|------|-------|
| Build group program structure | TG-001 (≥3 clients) | 4-6 week cohort, pricing TBD |
| Call prep automation | TG-002 (≥5 calls) | Rich can build pre-call brief from LinkedIn profile |
| Onboarding process | TG-003 (≥1 client) | Welcome email, first session template, progress tracking |
```

This is **not** task creation. It's a parking lot for strategic intent, gated by thresholds. When a gate unlocks, the deferred work entry becomes the seed for task creation. The idea is captured once, not re-invented.

**The enforcement mechanism is now:**
1. Gate is LOCKED → work stays in "Deferred Work" section
2. Daily planner checks gates → if LOCKED, items appear in "⏸️ Explicitly NOT Doing"
3. Gate fires (weekly check) → notification + item moves from Deferred to active path planning
4. Only THEN are tasks created

This is deferral with memory, not deferral through absence.

## 8. Summary of Recommendations

| # | Issue | Recommendation | Priority |
|---|-------|---------------|----------|
| 1 | Task IDs in goal file (Leak 2) | Remove task ID ranges from Paths table. Link by Path ID only. | HIGH |
| 2 | Bottleneck Action references tasks (Leak 3) | Reference paths, not task IDs. | MEDIUM |
| 3 | Daily plan data flow unspecified | Define two-layer input (goal context + task actions) and output format. | HIGH |
| 4 | Threshold gate firing mechanism absent | Specify: weekly check, notification, no auto-creation. Add `source` and `check frequency` to gate template. | HIGH |
| 5 | Stop criteria mostly unenforceable | Drop or defer unmeasurable criteria. Each criterion must be a query. | HIGH |
| 6 | No path retrospective on pivot | Add `Path Archive` section to goal file for dropped/paused paths with final metrics. | MEDIUM |
| 7 | No active deferral mechanism | Add `Deferred Work` section gated by thresholds. Daily plan surfaces "NOT doing" list. | HIGH |
| 8 | Shared Requirement coupling is necessary but needs maintenance rules | Document the update rule. Sync script should flag orphaned IMPLEMENTED_BY edges. | MEDIUM |
| 9 | System Status transitions ungated | Define enforceable gate conditions for BUILDING → OPERATIONAL → OPTIMISING. | MEDIUM |

---

## 9. Final Assessment

The v1.3 draft gets the hard part right: **the hierarchy is correct and the separation point (Path as bridge) is well-chosen.** Strategy is stable, execution is volatile, and you can pivot paths without touching goals.

The gaps are operational, not architectural:
- **How do gates fire?** (Unspecified)
- **How does the daily planner use both layers?** (Unspecified)
- **How does the system enforce deferral?** (Passive absence, not active mechanism)
- **Are stop criteria real or aspirational?** (Mostly aspirational)

These are solvable problems. The bones are solid. Fix the operational gaps and this framework is ready to implement.
