## 1. The Hierarchy

Five entity types, cleanly separated into two layers:

```
STRATEGY LAYER (why + what)
  Goal → Sub-Goal → Mechanism Step

EXECUTION LAYER (how)
  Path → Task → Subtask
```

**Goal** — A measurable business outcome with a deadline. One sentence, one number, one date.
*Example: "£5k/month recurring profit by 2026-06-30"*

**Sub-Goal** — A decomposed target that, combined with siblings, achieves the parent Goal. Still measurable.
*Example: "Close 6 clients @ £500+/month average"*

**Mechanism Step** — One stage in the conversion funnel that transforms an input quantity into an output quantity. Mechanism steps chain together with conversion ratios. This IS the Theory of Constraints surface — the step with the worst ratio or lowest throughput is the bottleneck.
*Example: "Speak to 100 people → 5 book calls → 1 closes" = 3 mechanism steps*

**Path** — An interchangeable channel/method that feeds a mechanism step. Paths are where execution lives. Each path has its own tasks, requirements, and conversion data.
*Example: LinkedIn outreach, Reddit engagement, Meetups — all feed "Speak to 100 people"*

**Task** — Existing v1.2 task files. Live inside a Path. No changes to task structure.

### Why This Separation Matters

- **Strategy layer** is stable. Goals and mechanisms rarely change. When they do, it's a strategic pivot.
- **Execution layer** is volatile. Paths get added/dropped/paused constantly. Tasks churn daily.
- You can swap all paths under a mechanism step without touching strategy.
- You can trace from any task up to the goal it serves in ≤4 hops.

## 3. Relationship Model

### Entity Relationships (What Connects to What)

```
Goal ──HAS_SUBGOAL──▶ Sub-Goal
Sub-Goal ──HAS_MECHANISM──▶ Mechanism Step
Mechanism Step ──CHAINS_TO──▶ Mechanism Step  (ordered: M1→M2→M3)
Mechanism Step ──FED_BY──▶ Path
Path ──CONTAINS_TASK──▶ Task
Task ──HAS_SUBTASK──▶ Subtask                (existing v1.2)
Task ──DEPENDS_ON──▶ Task                     (existing v1.2: hard/soft/threshold)

Goal ──REQUIRES──▶ Shared Requirement
Shared Requirement ──IMPLEMENTED_BY──▶ Task
Shared Requirement ──BLOCKS──▶ Mechanism Step

Goal ──HAS_THRESHOLD_GATE──▶ Threshold Gate
Threshold Gate ──UNLOCKS──▶ Task | Path | Sub-Goal
Threshold Gate ──READS_FROM──▶ Metric Source
```

### Key Design Decisions

1. **Mechanism Steps belong to Sub-Goals, not Goals directly.** A goal might have multiple sub-goals with different funnels.
2. **Paths feed Mechanism Steps, not Sub-Goals.** This keeps paths interchangeable at the right level.
3. **Shared Requirements are a distinct entity type** — not just tasks. They have their own status and block at the mechanism level. But they are IMPLEMENTED BY tasks. See §5 Q3 for rationale.
4. **Threshold Gates are first-class entities** with their own metric source, not just dependency annotations. This lets the scanner query all gates across all goals.

## 4. Neo4J Schema

### Node Types

```cypher
// Strategy Layer
(:Goal {
  id: String,          // "GOAL-001"
  name: String,
  metric: String,      // "£5000/month"
  metricValue: Float,  // 5000
  metricUnit: String,  // "GBP/month"
  measurement: String, // "Stripe MRR"
  deadline: Date,
  status: String,      // ACTIVE|PAUSED|ACHIEVED|ABANDONED
  systemStatus: String,// BUILDING|OPERATIONAL|OPTIMISING
  created: Date
})

(:SubGoal {
  id: String,          // "SG-001"
  name: String,
  metric: String,
  metricValue: Float,
  currentValue: Float,
  target: Float,
  status: String
})

(:MechanismStep {
  id: String,          // "M1"
  name: String,        // "Reach"
  inputLabel: String,  // "People contacted"
  outputLabel: String, // "Conversations started"
  conversionRatio: Float,  // 0.05
  currentThroughput: Float,
  throughputUnit: String,  // "per_week"
  isBottleneck: Boolean
})

// Execution Layer
(:Path {
  id: String,          // "PATH-LI"
  name: String,        // "LinkedIn Outreach"
  status: String,      // ACTIVE|PAUSED|NOT_STARTED|DROPPED
  isMVP: Boolean,
  requirements: [String], // path-specific reqs beyond shared
  created: Date
})

(:Task { ... })        // Existing v1.2 — no changes
(:Subtask { ... })     // Existing v1.2 — no changes

// Cross-cutting
(:SharedRequirement {
  id: String,          // "SR-001"
  name: String,        // "Coaching offer defined"
  status: String,      // COMPLETE|INCOMPLETE|IN_PROGRESS
  blocksStep: String   // Which mechanism step(s) this blocks
})

(:ThresholdGate {
  id: String,          // "TG-001"
  condition: String,   // "≥3 clients signed"
  metric: String,
  threshold: Float,
  currentValue: Float,
  source: String,      // Where to read the metric
  status: String       // LOCKED|UNLOCKED
})
```

### Edge Types

```cypher
// Strategy hierarchy
(:Goal)-[:HAS_SUBGOAL {order: Int}]->(:SubGoal)
(:SubGoal)-[:HAS_MECHANISM {order: Int}]->(:MechanismStep)
(:MechanismStep)-[:CHAINS_TO {order: Int}]->(:MechanismStep)

// Paths feed mechanism steps
(:Path)-[:FEEDS {contribution: Float}]->(:MechanismStep)

// Execution
(:Path)-[:CONTAINS_TASK {order: Int}]->(:Task)
(:Task)-[:HAS_SUBTASK]->(:Subtask)           // existing
(:Task)-[:DEPENDS_ON {type: String}]->(:Task) // existing

// Shared requirements
(:Goal)-[:REQUIRES]->(:SharedRequirement)
(:SharedRequirement)-[:IMPLEMENTED_BY]->(:Task)
(:SharedRequirement)-[:BLOCKS]->(:MechanismStep)

// Threshold gates
(:Goal)-[:HAS_GATE]->(:ThresholdGate)
(:ThresholdGate)-[:UNLOCKS]->(:Task | :Path | :SubGoal)
(:ThresholdGate)-[:READS_FROM {field: String}]->(:MetricSource)
```

### Key Queries This Enables

```cypher
// 1. What's blocking the goal right now?
MATCH (g:Goal {id:'GOAL-001'})-[:REQUIRES]->(sr:SharedRequirement {status:'INCOMPLETE'})
RETURN sr.name, sr.blocksStep

// 2. What's the current bottleneck?
MATCH (g:Goal)-[:HAS_SUBGOAL]->()-[:HAS_MECHANISM]->(m:MechanismStep {isBottleneck:true})
RETURN m.name, m.conversionRatio, m.currentThroughput

// 3. Trace from task to goal
MATCH (t:Task {id:'LP-003'})<-[:CONTAINS_TASK]-(p:Path)-[:FEEDS]->(m:MechanismStep)
      <-[:HAS_MECHANISM]-(sg:SubGoal)<-[:HAS_SUBGOAL]-(g:Goal)
RETURN t.id, p.name, m.name, sg.name, g.name

// 4. Which threshold gates are close to unlocking?
MATCH (tg:ThresholdGate) WHERE tg.currentValue >= tg.threshold * 0.8
RETURN tg.condition, tg.currentValue, tg.threshold

// 5. MVP path status — what's left?
MATCH (p:Path {isMVP:true})-[:CONTAINS_TASK]->(t:Task)
WHERE t.status <> 'COMPLETED'
RETURN t.id, t.name, t.status ORDER BY t.priority

// 6. All paths feeding the bottleneck
MATCH (p:Path)-[:FEEDS]->(m:MechanismStep {isBottleneck:true})
RETURN p.name, p.status
```

## 6. Critical Questions — Addressed

### Q1: Do inputs/outputs apply at sub-goal and path level, or only at task/subtask level?

**Answer: Both, but differently.**

- **Task/subtask level:** Inputs/outputs are concrete artifacts (files, data, states). This is existing v1.2.
- **Mechanism step level:** Inputs/outputs are quantities (people, conversations, calls). They have conversion ratios.
- **Sub-goal level:** Input = mechanism chain output. Output = metric toward goal. No separate I/O needed — the mechanism chain IS the I/O specification.
- **Path level:** No formal I/O. Paths contribute throughput to a mechanism step. Their "output" is just their share of the step's output quantity.

**Tradeoff:** You could add I/O to paths (e.g. "LinkedIn outputs qualified conversations"). I'd avoid it — it duplicates the mechanism step's output and creates maintenance burden. Track path contribution as a number on the FEEDS edge instead.

### Q2: How do you represent goal-level blockers (shared requirement status)?

**Answer: Shared Requirements are first-class nodes that BLOCK mechanism steps.**

They sit at the Goal level, not inside any path. They surface as a simple table in the goal file. In Neo4J, they're nodes with BLOCKS edges to MechanismSteps and IMPLEMENTED_BY edges to Tasks.

The scanner query is trivial:
```cypher
MATCH (g:Goal)-[:REQUIRES]->(sr {status:'INCOMPLETE'})-[:BLOCKS]->(m)
RETURN sr.name, m.name
```

### Q3: Are shared requirements themselves tasks? Or a different entity type?

**Answer: Different entity type, implemented by tasks.**

Rationale:
- A shared requirement is a *condition* ("offer must exist"), not work ("write the offer document").
- The *work* to satisfy the condition is a task (BC-OFFER-V1).
- One shared requirement might need multiple tasks (e.g. "legal compliance" = contract + T&Cs + privacy policy = 3 tasks).
- A shared requirement's status is derived from its implementing tasks' statuses.

**Do shared requirements have multiple paths?** No. They're atomic conditions. If there were truly multiple ways to satisfy one, you'd model that as multiple tasks implementing it (e.g. "payment system" could be implemented by "set up Stripe" OR "set up PayPal" — two tasks, one SR).

⚠️ **TRADEOFF FOR JAMIE:** This adds an entity type. Simpler alternative: just tag certain tasks as `shared: true` and infer blocking from their dependencies. Loses the clean "goal-level view" but avoids a new concept. I recommend the separate entity — it's only ~4-6 items per goal and the clarity is worth it.

### Q4: How does tracing work at strategic vs execution level?

**Answer: Two trace directions, clean boundary at Path.**

**Top-down (strategic):** Goal → Sub-Goal → Mechanism Step → "What's the bottleneck?" → Which paths feed it?
**Bottom-up (execution):** Task → Path → Mechanism Step → Sub-Goal → Goal → "Why am I doing this?"

The **Path** entity is the bridge between layers. Above it = strategy (stable, quantitative). Below it = execution (volatile, task-driven).

In Neo4J, any node can be traced to the goal in ≤4 hops:
```
Task → Path → MechanismStep → SubGoal → Goal
```

### Q5: How do you avoid over-engineering massive dependency chains while retaining full traceability?

**Answer: Three rules.**

1. **Strategy layer has NO dependencies.** Mechanism steps chain (M1→M2→M3) but this is ordering, not dependency. You don't "block" M3 because M2 isn't done — M2 is never "done," it's a continuous funnel.

2. **Cross-path dependencies are ONLY expressed through Shared Requirements.** If LinkedIn needs an offer and Meetups need an offer, that's one SR, not a dependency between paths. This prevents the combinatorial explosion.

3. **Dependencies live ONLY at task level (existing v1.2).** Hard/soft/threshold deps between tasks within and across paths. The goal layer doesn't add new dependency types — it adds context (why this task matters) and surfacing (what's actually blocking the goal).

**The insight:** The goal framework is a LENS on existing tasks, not a new dependency layer. It answers "what matters" and "what's stuck" without creating new chains to manage.

### Q6: How does this translate to Neo4J?

See §4 above. Every entity is a node, every relationship is an edge, every property is a property. The markdown tables map 1:1 to graph elements.

**Migration path:**
1. Goal files are markdown (human-readable, ADHD-friendly)
2. A sync script reads goal files → upserts Neo4J nodes/edges
3. Queries run against Neo4J for bottleneck analysis, tracing, dashboards
4. Humans interact with markdown; machines interact with graph

## 7. Uncertainties and Tradeoffs for Jamie's Input

### 🟡 Needs Jamie's Decision

1. **Mechanism step granularity:** I used 3 steps (Reach → Engage → Close). Should "Engage" split into "Conversation Started → Call Booked" as two steps? More granular = better bottleneck detection, but more to maintain. My recommendation: start with 3, split when you have data showing where in "Engage" people drop off.

2. **Sub-goal decomposition for SG-002 (expenses):** This is blocked on FIN-001. Does it belong in this goal or is it a separate goal ("Financial Stability")? Revenue goals and expense goals feel like they should be siblings under a parent goal, not nested. Could imply a "Meta-Goal" layer. I'd avoid that complexity — just make two goals that reference each other.

3. **Where do metrics live?** Goal files have current values in markdown tables. But for the scanner/Neo4J to work, these need a single source of truth. Options: (a) H1 sheet columns, (b) dedicated metrics file, (c) Neo4J properties updated by scripts. My recommendation: (a) for pipeline metrics (already there), (c) for goal-level metrics (updated weekly by cron).

4. **Path-level conversion tracking:** Should each path track its own conversion ratio at each mechanism step? (LinkedIn: 5% reach→engage vs Meetups: 15% reach→engage). This is powerful for comparing paths but adds tracking overhead. My recommendation: yes, but only for ACTIVE paths, and only after ≥20 data points per step.

### 🟢 Confident Decisions (Made Without Asking)

- Shared Requirements are a separate entity type (see Q3 rationale)
- No dependencies at strategy layer (see Q5 rule 1)
- Path is the bridge entity between layers
- System stop criteria are a checklist, not a scored metric
- Threshold gates are first-class entities, not just task dependency annotations

### 🔴 Known Gaps

- No template yet for Sub-Goal files or Path files (only Goal file templated). Do these need their own files or are they sections within the Goal file? My recommendation: keep them as sections in the goal file until a goal has >3 sub-goals or >5 paths, then split.
- No scanner/cron spec for reading threshold gates and updating statuses
- No migration guide from current task registry → goal-linked tasks
- Mechanism step ratios are estimates — need real data to calibrate
